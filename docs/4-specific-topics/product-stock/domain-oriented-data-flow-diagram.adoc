== External Entities

The following external entities interact with the Product Stock feature:

* Student – The primary actor of the feature. The student creates product stocks, views stock details, and manages individual stock items by adding, removing, or editing them.

== Processes

The following processes represent the core business operations of the Product Stock feature:

=== Process 1: Create Product Stock

The student provides a product selection and an item list. In return, the student receives a stock creation confirmation once the stock has been registered.

=== Process 2: View Product Stock

The student provides a stock view request indicating the product they want to inspect. In return, the student receives the product stock details.

=== Process 3: Add Item

The student provides new item information for the product stock they are currently viewing. The new item is then recorded in the Inventory.

=== Process 4: Remove Item

The student indicates the item they want removed from the product stock they are currently viewing. The item is then deleted from the Inventory.

=== Process 5: Edit Item

The student provides the item they want to edit along with the updated item information. In return, the student receives confirmation that the item was successfully modified.

== Data Stores

The following data store supports the Product Stock feature:

* D1 Inventory – Stores all product stock records and individual item entries. It is read and written by all five processes within this feature.

== Identified Risks and Considerations

The following risks were identified during the development of this diagram:

* Ambiguity in stock requirements may result in missing or incorrect data flows during implementation.
* The Add Item, Remove Item, and Edit Item processes each reference the same Inventory data store. These are represented separately in the diagram for readability but refer to a single shared store.
* Edge cases such as empty stock, failed retrievals, or partial updates are not represented in this domain-level diagram and should be addressed at the implementation level.
* The diagram intentionally excludes UI and technical implementation details in order to maintain a domain-oriented perspective.

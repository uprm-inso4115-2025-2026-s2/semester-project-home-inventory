== Core Prevention Principles

. Duplicate prevention must occur before database insertion.
. Barcode match is authoritative.
. Product definitions must be normalized before comparison.
. Near-duplicates require explicit user confirmation.
. Spend events must only be created after duplicate resolution is finalized.
. Prevention logic must be deterministic and reproducible.

== Duplicate Detection Hierarchy

The system must evaluate duplicates using the following strict hierarchy:

=== Level 1 – Barcode Match (Authoritative)

If a barcode (UPC/EAN) exists:

* Query Supabase for existing product with identical barcode.
* If found:
** Reuse existing Product ID.
** Do NOT create new Product definition.
** Apply stock duplicate handling rules.
* If not found:
** Create new Product definition.

Hard Rule:
Barcode match overrides name/brand differences.

=== Level 2 – Structured Metadata Match

If no barcode is provided:

Compare normalized:

* product_name
* brand
* size
* variant (optional)
* unit_of_measure

Matching Logic:

Exact match on:
`normalized_name + normalized_brand + normalized_size`

If exact match found:

* Suggest reuse of existing product.
* Default action: reuse.

If partial match (name + brand only):

* Treat as near-duplicate.
* Require confirmation.

=== Level 3 – Fuzzy / Similarity Check

If no exact structured match:

Perform similarity detection:

* Case-insensitive comparison
* Remove extra spaces
* Remove punctuation
* Normalize units (oz → ounce, lb → pound)

If similarity threshold > 85%:

* Flag as potential duplicate
* Show confirmation prompt

Default action: create new product unless user confirms reuse.

== Normalization Rules

All text fields must be normalized before comparison.

=== Text Normalization

* Convert to lowercase
* Trim leading/trailing whitespace
* Collapse multiple spaces into one
* Remove punctuation
* Replace accented characters
* Normalize common abbreviations:
** oz → ounce
** lb → pound
** ct → count

Example:

" Clorox  64oz.  Lemon "  
becomes  
"clorox 64 ounce lemon"

=== Size Normalization

Convert sizes to canonical format:

* Store numeric value separately from unit
* Convert compatible units (if possible)
Example:
64 oz → 64 ounce
1 lb → 16 ounce (optional advanced rule)

=== Brand Normalization

Brands must be:

* Lowercase
* Trimmed
* Stored consistently (no trailing spaces)

== Supabase Query Strategy

All duplicate checks must be executed BEFORE insert.

=== Barcode Check Query

[source,sql]
----
SELECT id FROM products
WHERE barcode = :barcode
LIMIT 1;
----

=== Structured Match Query

[source,sql]
----
SELECT id FROM products
WHERE normalized_name = :normalized_name
AND normalized_brand = :normalized_brand
AND normalized_size = :normalized_size
LIMIT 5;
----

=== Similarity Check (Optional Advanced)

If using Postgres full-text search:

[source,sql]
----
SELECT id, similarity(normalized_name, :normalized_name) AS score
FROM products
WHERE similarity(normalized_name, :normalized_name) > 0.85
ORDER BY score DESC
LIMIT 5;
----

Note:
Similarity search should NOT auto-merge — only suggest.

== Integration Notes (Flutter + Supabase)

=== Flutter Responsibilities

* Normalize fields before sending to backend.
* Show duplicate warning modals.
* Require explicit user confirmation for:
** near-duplicate reuse
** merge actions

=== Supabase Responsibilities

* Enforce unique constraint on barcode.
* Store normalized fields in dedicated columns.
* Execute duplicate queries atomically before insert.
* Reject conflicting barcode inserts.

Recommended DB Constraints:

* UNIQUE(barcode)
* INDEX(normalized_name, normalized_brand, normalized_size)

== Data Integrity Guarantees

The system must guarantee:

* Duplicate prevention decisions are auditable.
* Spend events only attach to confirmed Product IDs.

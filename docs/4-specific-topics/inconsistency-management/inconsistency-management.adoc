== Inconsistency Management

An exploration of our domain terminology, database schema, events, actions and behaviors, shared inventory definition, and duplicate item criteria led to the discovery of two inconsistencies.
* **Event naming clash — "An Inventory has been generated":** This event, triggered by the addition of an item to the inventory for the first time, was described as changing the inventory so that it now reflects the presence of a new item. However, the term _Inventory_ is defined in our terminology as "the list of available products in a home," while _Item_ (a synonym for _Stock_) is defined as "an instance of a product." The event description created a designation clash with the existing glossary: the name alluded to the creation of an inventory, but the described change was a modification of an already-existing inventory. To resolve this clash, the event was split into two:
** The "What changed" description of "An Inventory has been generated" was updated to: "The inventory is created and reflects the presence of the new item." This separates the creation of the inventory from the modification of an existing inventory, and clarifies that the inventory is created as a direct result of adding an item for the first time.
** A new event, **"A new Product has been generated"**, was introduced for cases where an item is added to the inventory with no relation to an existing product. This event records the change to the inventory reflecting the new product description as well as the item count and description.

* **Ownership multiplicity mismatch — Group-Owned Inventory model:** The Group-Owned Inventory model in the Shared Inventory Definition document describes how all items can be mapped as belonging to the household, with multiple members able to add, edit, and delete them. It further describes that inventories have multiple stock entries, and that stock entries may optionally carry linked owner tags. This implies that items can have multiple owners. However, the ER and table diagrams for database item representation did not reflect this: the model was only satisfiable when items had exactly one owner.
** To resolve this weak conflict, the ER and table diagrams for database item representation were updated. The relationship between items and owners was changed to a many-to-many relationship with total participation on both sides. A new table, **Owns**, was added to represent this relationship. This change allows an item to be owned by multiple members while preserving the ability of a member to own multiple items.

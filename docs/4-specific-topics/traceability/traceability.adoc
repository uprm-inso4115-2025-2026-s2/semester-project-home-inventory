== Traceability

This section describes how traceability is established in the Digital Home Inventory project. Traceability means our deliverables are linked so we can answer questions about where requirements come from, which personas they serve, and how they connect to domain concepts, scenarios, and verification artifacts.

=== Overview

Traceability helps us understand relationships between different parts of our documentation. When something changes, we can see what else might be affected. For example, if a requirement is modified, we can identify which personas and needs it supports, which user stories are impacted, and which scenarios/tests should be revisited.

Our approach focuses on creating clear links between artifacts without overcomplicating the documentation (Milestone 1 focuses on progress, not completeness).

=== Traceability approach (labels and links)

We maintain traceability through a simple labeling system:

==== Persona goals

Each persona’s goals are labeled with a short code. (Goal descriptions are kept short to make them easy to reference.)

* **Mateo (M)**
  * **MG1**: Know what I have without extra effort
  * **MG2**: Avoid wasting expired food
  * **MG3**: Avoid running out of daily essentials

* **Sofia (S)**
  * **SG1**: Stay within a strict budget
  * **SG2**: Split shared costs fairly
  * **SG3**: Avoid duplicate purchases

* **Julian (J)**
  * **JG1**: Avoid running out of specialty items
  * **JG2**: Avoid waste from bulk purchases
  * **JG3**: Keep personal items separate

* **Andrea (A)**
  * **AG1**: Coordinate shared stocking without mental overload
  * **AG2**: Get reimbursed accurately for shared purchases
  * **AG3**: Know what is needed before bulk shopping

==== Needs ↔ requirements

Each requirement in Section 2.2 is tagged with which Needs IDs (N1–N5) it supports, and (when helpful) which persona goals it supports.

.Example
* **IR-01 (Item Initialization)** → Needs: N1, N2 | Goals: MG1, JG1, AG3
* **IR-04 (Low-stock thresholds)** → Needs: N2 | Goals: MG3, JG1, AG1
* **IR-07 (Cost splitting)** → Needs: N5 | Goals: SG2, AG2

==== User stories ↔ requirements

Each user story in Section 2.2.1 lists which interface requirements it implements.

.Example
* **US-INV-01** → Implements: IR-01, IR-02, IR-03, IR-05
* **US-ALT-01** → Implements: IR-04
* **US-SB-01** → Implements: IR-06, IR-07
* **US-RPT-01** → Implements: IR-08

==== Domain concepts ↔ requirements

Domain concepts from Section 2.1 (Terminology/Entities/Events/Functions) are linked to the requirements that depend on them.

.Example
* **Item type / Item instance** → Appears in: IR-01, IR-02, IR-05
* **Household / Household member** → Appears in: IR-03, IR-06, IR-07
* **Low-stock threshold** → Appears in: IR-04
* **Expiration date** → Appears in: IR-05
* **Spending entry / Budget (optional scope)** → Appears in: IR-06, IR-07, IR-08

==== Scenarios/tests ↔ requirements (Milestone 1)

For Milestone 1, traceability is extended to the main validation and verification artifacts:

* Scenario IDs: **S1..Sx** (Section 3.2.1)
* Test case IDs: **TC-*** (Section 3.2.3)
* Walkthrough findings: **WF-*** (Section 3.2.2) once real sessions are performed

.Example
* **S1 (Low-stock after consumption)** → IR-02, IR-04 → TC-ALT-01, TC-AUTO-01
* **S2 (Avoid duplicate purchase in shared household)** → IR-03, IR-08 → TC-INV-05, TC-RPT-01

=== Questions traceability helps answer (kept concise)

* Which personas and needs are affected if a requirement changes?
* Which scenarios/tests must be revisited to verify the changed requirement?
* Which domain concepts are involved in a given requirement (to avoid terminology drift)?

=== Current traceability status (Milestone 3)

We have established traceability for:

* Persona goals: all personas have labeled goals (MG*, SG*, JG*, AG*).
* Needs: core needs N1–N5 are referenced by interface requirements and M3 refinements.
* Requirements: IR-01 through IR-14, DR-1 through DR-25, and MR-01 through MR-08 are linked to needs, concepts, or Milestone 3 acquisition evidence where applicable.
* Validation/verification: key scenarios and test cases are linked to the most central requirements.
* Milestone 3 acquisition evidence: each requirement added or revised during Milestone 3 references a specific M3-QA source in Section 4.4.5.

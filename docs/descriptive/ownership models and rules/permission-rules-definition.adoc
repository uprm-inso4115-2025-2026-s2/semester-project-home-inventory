= Permission Rules Definition

== Objective
Establish clear role-based permission rules so system actions are consistent and prevent unauthorized changes.

== Roles Model

=== Primary Roles

The primary roles in the application are:

- Owner: Oftentimes the creator of the household, with full permissions to manage the household and its members. Owners can invite/remove members, change permissions, and delete the household.
- Member: A member of the household with permissions to interact with shared inventory. Members lack administrative capabilities such as managing permissions, but can send invitation and add/edit/ delete inventory items with the right permissions.
- Viewer: A read-only role that can view shared inventory but cannot make changes or send invitations.

=== Role Hierarchy

The hierarchy is as follows (from highest to lowest):

1. Owner
2. Member
3. Viewer

All permissions of a lower role are included in the higher role. For example, an Owner has all the permissions of a Member and a Viewer, while a Member has all the permissions of a Viewer.

=== Membership States

- Active: The user is an active member of the household and has the permissions associated with their role.
- Removed: The user has been removed from the household and no longer has access to any resources or permissions.
- Pending Invite: The user has been invited to join the household but has not yet accepted the invitation. They do not have any permissions until they accept the invite and become an active member.

== Actions Catalog

=== Actions
- Inventory: Create / Read / Update / Delete
- Members: View / Remove / Leave
- Invites: Create / View / Revoke / Accept / Decline
- Household Settings: View / Update (name, settings toggles, etc.)
- Role Management: Promote / Demote / Transfer Ownership


=== Owner Rules

Owner rules include:
- Full access to all resources and actions within the household.
- Can manage members and permissions, including inviting new members, removing existing members, and transferring ownership.
- Can manage inventory items, including creating, editing, and deleting any item in the shared inventory.
- Can manage household settings, such as changing the household name or toggling settings.      


=== Member Rules

Member rules include:
- Read and write access to inventory items within the household.
- Ability to view and manage their own membership details.
- Ability to leave the household.

=== Viewer/Guest Role Rules

Viewer rules establish that viewers can only view shared inventory but cannot make changes or send invitations.

== Inheritance, Overrides, and Configurability

=== Permission Inheritance

Permissions are inherited based on role hierarchy.
Higher roles automatically have all permissions of lower roles.

=== Overrides / Exceptions

Exceptions with item edit permissions can occur when using the User-Owned Within Group ownership model, where only the creator of an item can edit it, regardless of their role.
- In the Group-Owned Inventory model, all members with the right permissions can edit items regardless of who created them.
- In the User-Owned Within Group model, only the creator of an item can edit it, regardless of their role.


=== Static vs Configurable Permissions

Hard-coded permissions are simpler to implement and ensure consistency across households, while configurable permissions offer flexibility but are more difficult to implement. For the Minimum Viable Product (MVP), we can implement static roles with predetermined permissions to ensure a clear and consistent user experience.

== Conflict Resolution Rules

=== Conflict Types

Conflict may arise when a user has multiple roles or when permissions from different rules overlap. For example, if an Owner wants to delete an item created by a Member, but the Member has edit permissions that allow them to prevent deletion, this creates a conflict.

Multiple invitation issued by different Owners for the same user, or when a user has both Owner and Member roles in different households that grant conflicting permissions.

=== Conflict Handling Policy

To handle conflicts, we can implement the following policies:

- Precedence of most restrictive permission: To prevent edit conflicts, we can implement a policy where the most restrictive permission takes precedence. For example, if a Member has edit permissions on an item but the Owner wants to delete it, the Owner's action would take precedence and the item would be deleted. However, if the Member has edit permissions that allow them to prevent deletion, then the deletion would be denied.

- Only one active Inivite: If a user receives multiple invitations from different Owners for the same household, the system should allow the user to accept only one invitation. Once an invitation is accepted, any other pending invitations for that user and household should be automatically revoked.

== Escalation & Transfer Scenarios

=== Ownership Transfer

- Preconditions: The user initiating the transfer must be the current Owner of the household, and the target user must be an active Member of the same household. The Member must also accept the ownership transfer invitation for the transfer to be successful.

- Steps: 
    1. The Owner initiates the transfer by selecting a Member and sending an ownership transfer invitation. 
    2. The Member receives the invitation and must accept it to complete the transfer. 
    3. Once accepted, the previous Owner's role changes to Member, and the new Owner gains all permissions associated with the Owner role.

- Resulting permissions changes: The previous Owner loses their Owner permissions and is demoted to a Member, while the new Owner gains all permissions associated with the Owner role.

- Transfer failure: If an Owner transfers ownership to a Member, the previous Owner becomes a Member and the new Owner gains all permissions. If the transfer fails mid-way, the system should ensure that the original Owner retains their permissions and the target Member does not gain any permissions until the transfer is completed.

=== Promotion / Demotion

Only Owners can promote or demote Members. The allowed transitions are as follows:

- Member to Owner: Allowed through ownership transfer process, which requires acceptance by the target Member.
- Owner to Member: Allowed by the current Owner, but the Owner must transfer ownership to another Member first before being demoted.
- Viewer to Member: Allowed by the Owner, but the Viewer must accept the role change.
- Member to Viewer: Allowed by the Owner, but the Member must accept the role change.

=== Removing Users

Only Owners can remove Members or Viewers from the household. When a user is removed, they lose all access to the household and its resources. The system should also ensure that any pending invitations for the removed user are revoked.

== Milestone 2 Target Deliverables

By the end of Milestone 2 (February 28 – April 3), the **Reports and Insights module** must be fully implemented and demo‑ready, with all features originally planned for Milestone 1 now delivered in this milestone.

The feature should allow a user to:

View a **dashboard (Home)** with:

  * Summary metric cards (inventory value, total items, expenses)
  * Interactive charts: pie chart (distribution of expenses by category), bar chart (items per room)
  * Dynamic filters by category, room, and date range
  * Real‑time data from Supabase (no mocks)

Access a **Reports section** with:

  * Detailed report views with line charts (expense history per month) and stacked bar charts (expense comparison by roommate)
  * Search functionality with real‑time results
  * Drill‑down capability: tap on chart elements to see detailed item lists
  * PDF export with metadata (date, filters applied)

Benefit from **backend optimizations**:

  * Materialized views for fast aggregated queries
  * Consumption‑rate functions (daily consumption, cost per use, inventory turnover)

* Use a library of **reusable UI components** (MetricCard, FilterChipBar, DateRangePicker, EmptyStateWidget) for consistency
* (Stretch) Experience a first‑time **onboarding flow** and PNG image export if time permits

All functionality must be built following Clean Architecture principles, connected to live Supabase data, and styled according to the existing high‑fidelity designs.

== High‑Level Implementation Scope

=== 1. Presentation Layer (Flutter UI):


* Dashboard (Home) Screen with filters and charts
* Reports Listing Screen
* Detailed Report Screens (with charts, tables, drill‑down)
* Search UI (SearchDelegate or custom search bar)
* PDF Export button and generation UI feedback
* Loading, Error, and Empty States for all screens
* Reusable component library extracted from common patterns

=== 2. State Management (Part of the Presentation Layer):

**DashboardCubit**:

  * fetchMetrics()
  * applyFilters()
  * clearFilters()

**ReportsCubit**:

  * fetchReports()
  * searchReports()
  * exportToPDF()

**DrillDownCubit** (for detail views):

  * fetchFilteredItems(category, room, etc.)

Well‑defined states:

  * Initial
  * Loading
  * Loaded (with data)
  * Error
  * Empty (no results)

=== 3. Domain Layer:

Entities:

  * `Metric` (for dashboard cards)
  * `Report` (for report listings)
  * `ChartDataPoint` (for all chart types)
  * `Filter` (to encapsulate filter state)

Repository interfaces:

  * `DashboardRepository`
  * `ReportsRepository`

(Optional but recommended) Use Cases:

  * `GetDashboardMetrics`
  * `GetReports`
  * `ExportReportAsPDF`

=== 4. Data Layer:

* **SupabaseDashboardRepository** implementation
* **SupabaseReportsRepository** implementation
* Materialized view queries (pre‑aggregated data)
* PostgreSQL function calls for consumption metrics
* Model definitions and JSON ↔ Entity mapping
* Caching strategy using Hive or SharedPreferences
* PDF generation service abstraction

=== 5. Backend / Database (Supabase/Postgres):

Materialized views:

  * `valor_inventario_por_categoria`
  * `gastos_por_mes`
  * `valor_por_habitacion`

PostgreSQL functions:

  * `calcular_tasa_consumo_diario`
  * `calcular_costo_por_uso`
  * `calcular_rotacion_inventario`

- Indexes for performance optimization
- Scheduled refresh strategy for materialized views (if needed)

=== 6. UI Styling & UX:

* Consistent spacing, typography, and color palette (already designed)
* Interactive chart tooltips and legends
* Form validation for filters (date ranges, etc.)
* Error handling with Snackbars/Dialogs
* Responsive layout for different screen sizes
* Pull‑to‑refresh for manual data refresh
* "Last updated" timestamp indicator

== Dependencies / Risks

=== Dependencies

* **Supabase schema finalized** – The underlying inventory and expense tables must be stable before materialized views can be created

* **Authentication system** – User ID must be available for row‑level security and personalized data

* **Charting library compatibility** – Must confirm `fl_chart` or chosen library works with current Flutter version

* **PDF generation library** – Must evaluate performance with complex charts (screenshot approach may be needed)

* **Design assets** – Designs are complete, but any missing icons or illustrations must be exported before UI implementation begins

=== Risks

[cols="1a,2a", options="header"]
|===
| Risk | Mitigation
| **Large data volumes** causing slow queries | Materialized views and proper indexing; implement caching layer
| **Chart performance** with many data points | Optimize data aggregation; test with realistic datasets early
| **PDF rendering of charts** | Evaluate screenshot‑based capture as fallback; prototype early
| **Storage permissions** on Android 10+ | Thorough testing on multiple API levels; use `permission_handler` correctly
| **Timeline compression** (all Milestone 1 tasks now in one milestone) | Strict prioritization; defer stretch goals if needed; focus on core deliverables first
| **Row‑Level Security (RLS)** blocking queries | Test with authenticated user early; involve backend specialist
| **Scope creep** | Team lead to enforce scope boundaries; any additions must go through change request
|===

=== Feature 1: Core Inventory Management (F-INV)

**User Value:** Eliminates tedious manual data entry by enabling quick item capture via barcode scanning with fallback manual entry. Reduces onboarding friction for household managers like Andrea.

**Primary Users:** Household managers, kitchen coordinators

**End-to-End User Flow:**

.Input / Process / Output Flow
[cols="2,3,2", options="header"]
|===
| **Input**
| **Processing**
| **Output**

| User taps "Add Item" floating action button
| System opens camera intent and initializes barcode scanner UI
| Camera preview displayed with scanning overlay

| User scans product barcode OR manually enters product name
| - Barcode detected: query product database (local cache + remote API)
- Product found: auto-populate (name, brand, image, default quantity)
- Product not found: show manual entry form fallback
| Item details form pre-filled or blank for manual entry

| User sets expiration date, quantity, location, and taps "Save"
| - Data validation (expiration date ≥ today, quantity > 0)
- Insert into Supabase `products` table
- Trigger Bloc/Cubit state update
- Stream listener broadcasts real-time update to inventory list
| Item appears instantly in Core Inventory list; list view refreshes via Flutter Bloc state management

| User views inventory list (sorted by category/status)
| Supabase real-time stream continuously syncs inventory state across all connected listeners
| Live inventory display with current quantities, expiration status, and item metadata

|===

**Detailed User Scenario (Andrea's Onboarding):**
1. Andrea opens the app and navigates to the "Inventory" tab
2. She taps the floating action button (FAB) to add her first item
3. Camera opens; she scans a pasta box
4. System recognizes it as "Barilla Penne Rigate" with product image
5. Auto-populated form shows: name, brand, category, image
6. She adjusts quantity to 2 boxes, sets expiration date to 2026-12-31
7. She taps "Save" and the item appears in the inventory list

**Components Involved:**
- **Presentation:** `AddItemPage`, `InventoryPage`, `ItemForm` widgets
- **State Management:** `InventoryCubit`, `InventoryState`
- **Data Layer:** `ProductSupabaseDataSource`, `InventorySupabaseDataSource`, `ProductRepositoryImpl`
- **Domain:** `AddInventoryItem`, `GetInventoryItems` use cases
- **External:** Barcode scanner plugin, Supabase Realtime streams

=== Feature 2: Automated Expiration Alerts & Notifications (F-ALERT + F-AUTOCHECK)

**User Value:** Prevents food waste by proactively alerting users before items spoil. Removes the mental burden of manual fridge checks, especially critical for busy users like Mateo.

**Primary Users:** All household members (push notifications), household admin (alert configuration)

**End-to-End User Flow:**

.Alert Generation & Delivery Pipeline
[cols="2,3,2", options="header"]
|===
| **Input**
| **Processing**
| **Output**

| Supabase Cron job triggers daily at 08:00 UTC
| 1. Query all non-archived items in `products` table
2. Calculate days until expiration for each item
3. Apply alert thresholds (e.g., notify if expires in ≤ 3 days)
4. Deduplicate: skip items already notified in last 24h (check `alert_log`)
5. Build payload with expiring items
| List of items meeting alert criteria, ready for notification

| Alert payload generated (item name, expiration date, action URL)
| 1. Validate user FCM/APNs token availability
2. Format payload with deep link to Alerts Center
3. Send via Firebase Cloud Messaging (FCM) or Apple Push Notification service (APNs)
| Push notification delivered to user's device

| User taps push notification on mobile device
| App route handles deep link and navigates to the "Alerts Center" tab to highlight expiring items
| Alerts Center screen displays:
- List of expiring items sorted by urgency
- Recommended actions (consume, donate, discard, extend date)
- Quick action buttons

| User taps "Mark as Consumed" or "Extend Expiration" for an item
| 1. If Mark Consumed: delete or reduce quantity to 0
2. If Extend: show date picker, update expiration date
3. Trigger state update via Bloc
4. Log action in `alert_log` to prevent duplicate notifications
| Item removed from or updated in inventory; alert cleared; confirmation shown

|===

**Detailed User Scenario (Mateo's Morning Alert):**
1. Mateo receives a push notification: "2 items expiring today or tomorrow"
2. Notification shows: "Milk (expires today)", "Yogurt (expires tomorrow)"
3. He taps the notification and the app opens and navigates to Alerts Center
4. He sees:
   - **Milk** | Expires: Today | Action: Consume Now / Discard
   - **Yogurt** | Expires: Tomorrow | Action: Consume Today / Extend to [date]
5. He taps "Consume Now" next to Milk and the item is marked consumed with quantity set to 0
6. For Yogurt, he taps "Extend" and the date picker appears; he extends it to 2 days out
7. Alerts list auto-refreshes; both items are now off the alert list

**Components Involved:**
- **Backend:** Supabase Cron Edge Function (TypeScript), Postgres triggers
- **Frontend Presentation:** Alerts Center screen, push notification handler
- **State Management:** Alert Cubit (if implemented) or real-time stream listener
- **Data Layer:** Alert repository, alert log data source
- **Domain:** `AutomatedCheckExpiration`, `AutomatedCheckStock` use cases
- **External:** Firebase Cloud Messaging (FCM), Apple Push Notification (APNs)

**Visual Evidence:**
[cols="1a,1a"]
|===
| image::screen-designs/Alerts-&-Notifications-Screen/push-notifications-flow-ui.png[Push Notification Flow, width=100%, title="Push Notification & Deep Link"]
| image::screen-designs/Alerts-&-Notifications-Screen/alerts-center-mockup.png[Alerts Center UI, width=100%, title="Alerts Center Screen"]
|===

**Key Implementation Details:**
- **Deduplication Logic:** Alert log stored in Supabase prevents notification spam; same item is not re-notified within 24 hours unless status changes
- **Threshold Flexibility:** Thresholds are configurable per household (e.g., some families want 7-day warnings, others 1-day)
- **Timezone Handling:** Cron job runs in UTC; expiration date comparison uses user's local timezone
- **Offline Handling:** If user is offline, Android/iOS queues notification locally; it displays when device reconnects

**Completion Status:** Completed — Alert generation logic, deduplication, and notification dispatch fully implemented. Alerts screen integrated with deep-link handling.

=== Feature 3: Smart Grocery List Management (F-GL-AUTO + F-GL-USER)

**User Value:** Closes the inventory lifecycle loop. Auto-generates shopping lists based on inventory state, then enables quick shopping trip management (check off items, restock on return).

**Primary Users:** Shoppers, household managers

**End-to-End User Flow:**

.Grocery List Generation & Trip Management
[cols="2,3,2", options="header"]
|===
| **Input**
| **Processing**
| **Output**

| Automated daily check (from Feature 2) runs and identifies inventory gaps
Example: Milk quantity = 0, Bread expired 2 days ago, Pasta stock below threshold
| 1. Process AutoCheck output (items with status: OUT_OF_STOCK, EXPIRED, LOW_STOCK)
2. Apply priority order:
   - Priority 1: OUT_OF_STOCK items
   - Priority 2: EXPIRED items
   - Priority 3: LOW_STOCK items
   - Priority 4: Frequently used + low/out
3. Skip items already on active grocery list
4. Log auto-additions to `grocery_list_log`
| Grocery list auto-updated with prioritized items (e.g., Milk [OUT_OF_STOCK], Bread [EXPIRED])

| User navigates to "My Grocery List" screen
| App fetches active grocery list from Supabase; displays items grouped by category
Options shown:
- Checkbox to mark item as "in cart"
- Edit quantity / notes
- Remove item
- Add custom item (manual entry)
| Grocery list displayed with auto-generated items + any user-added custom items

| User is at supermarket; checks off items as they enter cart
Then, user completes shopping trip and taps "Finalize Grocery Run"
| 1. For each checked item:
   - Prompt for quantity bought (default = inventory gap amount)
   - Prompt for expiration date (if perishable)
2. Collect receipt info (optional: total spent, payment method)
3. Validate entries
| Dialog/form shown for final confirmation + quantity/expiration entry

| User confirms and taps "Save & Restock"
| 1. For each item, update/insert into Core Inventory with new quantities and expiration dates
2. Clear checked items from grocery list (or archive)
3. Log restocking action in `grocery_list_log`
4. Trigger inventory state update (Bloc)
| Updated inventory displayed in Core Inventory screen; grocery list cleared of checked items; confirmation message shown

|===

**Detailed User Scenario (Andrea's Weekly Shopping Trip):**
1. Andrea opens the app on Sunday; she sees the Grocery List tab shows 6 auto-generated items:
   - Milk (OUT_OF_STOCK) — Qty: 1
   - Bread (EXPIRED) — Qty: 1
   - Pasta (LOW_STOCK) — Qty: 2
   - Eggs (LOW_STOCK) — Qty: 3
   - Cheese (added manually by Andrea)
   - Butter (added manually by Andrea)
2. Andrea heads to the supermarket
3. She checks off items as she places them in her cart:
   - Milk (1 liter)
   - Bread (2 loaves)
   - Pasta (2 boxes)
   - Eggs (1 dozen)
   - Cheese (500g)
   - Butter (500g)
4. At checkout, she taps "Finalize Grocery Run"
5. System shows a summary dialog:
   ```
   Review your purchases:
   - Milk [1 box] | Expires: 2026-04-20
   - Bread [2 loaves] | Expires: 2026-04-08
   - Pasta [2 boxes] | Expires: 2027-04-07
   - Eggs [1 dozen] | Expires: 2026-05-01
   - Cheese [500g] | Expires: 2026-06-15
   - Butter [500g] | Expires: 2026-05-01
   [Spent: $--]  [Save & Restock]
   ```
6. She taps "Save & Restock"
7. App updates Core Inventory with new items and quantities
8. Grocery list is cleared (checked items archived)
9. Confirmation: "6 items restocked! Your inventory is up to date."

**Components Involved:**
- **Presentation:** `MyGroceryListPage`, `GroceryListTile`, `AddCustomItemPage` widgets
- **State Management:** `GroceryListCubit` (if implemented) or stream-based updates
- **Data Layer:** Grocery list repositories and data sources, expense tracking
- **Domain:** Grocery list use cases for generation and update
- **Auto-Generation:** Scheduled function that runs daily AutoCheck output processing

=== Feature 4: Shared Inventory & Household Management (F-SHARING)

**User Value:** Enables multiple household members to collaborate on inventory and see shared items, roles, and responsibilities.

**Primary Users:** All household members

**Current Status:** 
- Household creation & member invitation: Implemented

- Shared vs. personal items: In Progress

**End-to-End User Flow (As Currently Implemented):**

.Household Setup & Member Invitation
[cols="2,3,2", options="header"]
|===
| **Input**
| **Processing**
| **Output**

| User creates a new household during onboarding
User enters: Household name, description
| 1. Validate input (non-empty name, max 100 chars)
2. Insert into `households` table with user as owner
3. Create initial role assignment (user = OWNER)
4. Initialize household settings (notification preferences, etc.)
| Household created; user becomes owner; household ID returned

| Owner navigates to Settings and selects Invite Members
Owner enters: Roommate email, role (MEMBER, ADMIN, VIEWER)
| 1. Generate unique invite token
2. Send email with invite link + token
3. Store invite record in `invitations` table
4. Log invitation action
| Invite email sent to roommate with deep link to accept/decline

| Roommate receives email and taps invite link
App opens to invitation screen showing: Household name, owner, role, permissions summary
Roommate taps "Accept"
| 1. Validate invite token (not expired, not already used)
2. Create user in `household_members` table with provided role
3. Mark invitation as accepted
4. Trigger Bloc update for household members list
| Roommate added to household; can now see shared items; confirmation shown

|===

- **Presentation:** Household settings, invite screen, member list view

=== Architecture & Technical Stack

**Frontend Stack:**
- **Framework:** Flutter 3.29.0 with Dart 3.7.0
- **State Management:** Flutter Bloc (Cubit for simple states)
- **Navigation:** GoRouter with deep-link support
- **UI Components:** Material 3 design, custom Material/Cupertino widgets
- **Local Storage:** Shared Preferences (settings), cached product data
- **Notifications:** Firebase Cloud Messaging (FCM) + platform-specific handling

**Backend Stack:**
- **Database:** Supabase PostgreSQL with Realtime subscriptions
- **Authentication:** Supabase GoTrue (email/password, social)
- **Backend Functions:** Supabase Edge Functions (TypeScript) for cron jobs
- **File Storage:** Supabase Storage (product images, receipts)
- **APIs:** Barcode lookup service (for product data enrichment)

**CI/CD & Deployment:**
- Version control: Git + GitHub
- Testing: Dart test framework, mocktail for mocking
- Linting: flutter_lints, Dart analyzer

=== Testing & Verification Strategy

Each completed feature has been verified through:

1. **Unit Tests:** Business logic (use cases, repositories) with mock data sources
2. **Widget Tests:** UI components in isolation (forms, lists, dialogs)
3. **Integration Tests:** End-to-end flows on emulator/physical device
4. **Manual QA:** Cross-device testing (Android, iOS), network conditions (offline, slow)

Example verification for F-INV (Inventory Management):
- Add item via barcode scanning
- Add item via manual entry fallback
- Edit existing item (quantity, expiration, category)
- Delete item
- Real-time list refresh on other device when item added
- Validation (no negative quantities, future expiration dates)

=== Known Limitations & Future Work

**Current Limitations:**
1. Role-based item visibility not yet enforced (all shared items visible to all members)
2. Budget alerts not yet implemented (spending-based notifications)
3. Complex searches (filter by location, owner, expiration range) deferred to post-MVP
4. Barcode scanning limited to common products; many items require manual entry

**Future Features (Post-MVP):**
1. **AI-Powered Meal Planning:** Suggest recipes based on current inventory
2. **Smart Notifications:** ML-based expiration risk scoring (some items spoil faster than labels suggest)
3. **Inventory Forecasting:** Predict when items will run out based on consumption patterns
4. **Family Analytics:** Who consumes what, eating pattern insights
5. **Integration with grocery delivery services:** Direct ordering from grocery list

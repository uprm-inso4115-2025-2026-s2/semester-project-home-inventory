== Software Design

Design decisions at a more detailed level than architecture (modules, domain model, APIs, etc.).

=== Key design decisions

* **Decision 1: Use Supabase Realtime for live synchronization**

Supabase Realtime will be used to propagate inventory and budget updates instantly to connected clients. This decision was mainly made because the application requires the roommates to see changes like item additions, deletions, budget updates, etc. with minimal delay. Realtime's Postgres Changes and Broadcast features will reduce the need for manual polling and improve the overall user experience while integrating seamlessly with the existing Supabase stack.

* **Decision 2: Enforce access control using Supabase Row-Level Security (RLS)**

All data access permissions will be enforced at the database level using the RLS Policies. This ensures that users will only see items belonging to their household. Using RLS reduces backend complexity, prioritizing security and preventing any data leaks. Policies will rely on `auth.uid()` and `household_members` to determine authorization.

* **Decision 3: Use Edge Functions for complex business logic**

Supabase Edge Functions will handle operations that require transactional guarantees or cross-table validation such as budget enforcement, invite validation, or conflict resolution. RLS alone will be insufficient for these cases, so Edge Functions will provide serverside execution in a secure Deno Runtime to allow small operations using database transactions.

=== Domain model notes

The following mappings describe how domain concepts translate into system components. Where the database table name differs from the domain term, both are shown.

[cols="2,2",options="header"]
|===
|Domain term |Implementation table name
|Household member |`users`
|Household |`households`
|Membership |`household_members`
|Location |`rooms` 
|Item type |`products` 
|Item instance  |`stock` 
|Restock list / planner |`shopping_lists` + `shopping_list_entries`
|Spending entry |`expenses`
|Budget |`budgets`
|Invitation |`invite_tokens`
|===

* **User (Household member)**
  - Source: Supabase Auth
  - Key field: `auth.uid()`
  - Related table: `users`
  - Purpose: ownership and identity

* **Household**
  - Table: `households`
  - Represents a shared living group
  - Links multiple users together

* **HouseholdMembership (Membership)**
  - Table: `household_members`
  - Fields: `user_id`, `household_id`, `role`
  - Used by RLS policies to authorize access
  - Supports admin vs member roles

* **Location** 
  - Represents a storage place within a household where inventory items may be stored
  - Associated with a Household through the `home_id` field
  - Examples include kitchen cabinet, pantry, fridge, and bathroom closet

* **Item type** 
  - Represents a reusable item type definition independent of a specific purchase
  - Used to normalize item metadata such as brand, size, and unit

* **Item instance / package** 
  - Represents a concrete unit of an item type tracked within the inventory
  - Includes quantity, status, purchase details, and expiration data
  - Real-time updates enabled via Supabase Realtime
  - Protected by RLS household policies

* **Item instance location** 
  - Represents the placement or movement of item instances within locations
  - Supports tracking of current location and location history

* **Inventory**
  - Table: `inventories`
  - Represents the collection of item instances associated with a household

* **Restock list / planner** 
  - Represents planned purchases for restocking household inventory
  - Associated with a restock list entry through the `shopping_list_entries_id` field

* **Restock list entry** 
  - Represents individual items within a restock list including quantity and priority

* **Category / Supplier**
  - Tables: `categories`, `suppliers`
  - Provide organizational and sourcing metadata for item types

* **Budget / Spending entry**
  - Tables: `budgets`, `expenses`, `household_summary`
  - Validated through Edge Functions before writes
  - Updated automatically via database webhooks

* **Invitation** 
  - Generated via Edge Function
  - Short-lived signed token
  - Used by `/accept-invite` endpoint

==== Realtime Channels

* `inventory-updates`
  - Purpose: notify clients when items change
  - Mechanism: Supabase Realtime Postgres Changes or Broadcast

* `budget-updates`
  - Purpose: push household budget recalculations

---

==== Edge Function Endpoints

* **POST /validate-budget** 
** Input: `{ amount, household_id }`
** *Behavior:*
    *** Computes current monthly spending
    *** Rejects if limit exceeded (403)
    *** Otherwise approves write
** Used before inserting expenses

* **POST /accept-invite**
** Input: `{ token }`
** *Behavior:*
    *** Validates signed invite token
    *** Adds user to `household_members`
    *** Returns success/failure

---

==== RLS Policy Strategy

* SELECT policies filters by household membership
* INSERT policies verifies user belongs to household
* Admin-only actions use helper function:
  - `is_household_admin(h_id uuid)`
* Private items enforced via `owner_id = auth.uid()`

== Machine Requirements

Machine requirements define measurable quality properties for Milestone 2 implementation. Since the prototype is under development, these requirements establish targets that will be verified upon completion. Where specific thresholds are not yet known, the requirement defines what will be measured and how success will be evaluated.

Each requirement includes traceability to relevant Domain Requirements (DR) where applicable.

=== MR-01: Availability

The system shall remain operational during supervised usability testing. No unplanned downtime requiring restart shall occur during testing sessions.

*Trace*: DR-12 (threshold-driven low-stock), DR-13 (expiration status)

*Verification*: Session logs; test coordinator observation.

---

=== MR-02: Response Time

The system shall respond quickly enough that household members can complete inventory tasks without waiting. During usability walkthroughs:
- No participant shall abandon a task due to perceived slowness
- All participants shall complete lookup tasks while simulating shopping (standing, holding phone)
- Item creation shall be completable in under 1 minute (consistent with US-INV-01)

Response time measurements will be collected during walkthroughs to establish baseline for future milestones.

*Trace*: DR-7 (short-window duplicates), DR-12 (threshold-driven low-stock)

*Verification*: Usability walkthrough observations; timing recorded for each task.

---

=== MR-03: Offline Operation

The mobile client shall support viewing inventory while offline and shall allow pending item creation and quantity updates while disconnected. Pending changes shall be clearly marked in the interface until synchronization succeeds. If the same tracked inventory record is changed remotely before synchronization, the system shall surface the conflict instead of silently overwriting data.

*Trace*: DR-9 (Auditability of corrections), DR-14 (deduplicated alerting), DR-25 (Pending offline inventory changes)

*Milestone 3 source*: M3-QA-08.

*Verification*: Integration tests with connectivity simulation; manual tests creating and updating items offline, then reconnecting.

---

=== MR-04: Synchronization

When multiple household members use the same inventory, changes shall be propagated to all members in real time via Supabase Realtime. Changes shall appear without requiring manual refresh.

*Trace*: DR-4 (household administration), DR-5 (membership independence)

*Verification*: Automated tests verifying real-time delivery; manual tests with 2-3 concurrent users confirming changes appear instantly.

---

=== MR-05: Security and Privacy

Only authorized household members shall have access to household inventory data. Personal item records shall be hidden from non-owners by default unless explicitly shared. Household role permissions shall be enforced consistently across inventory, alerts, spending, and membership-management views.

*Trace*: DR-4 (household administration), DR-5 (membership independence), DR-6 (attribution retention), DR-18 (Personal item default privacy), DR-19 (Role-scoped household permissions)

*Milestone 3 source*: M3-QA-01, M3-QA-02.

*Verification*: Access control tests using owner/member/viewer/admin test accounts.

---

=== MR-06: Auditability

Quantity changes shall be logged with timestamp, user ID, item, and previous value.

*Trace*: DR-8 (ambiguity guard), DR-9 (auditability of corrections)

*Verification*: Schema validation; manual log inspection.

---

=== MR-07: Notification Quality

Repeated automated checks without meaningful state changes shall not generate duplicate alert records or repeated push notifications. Shared household alerts shall support household-level in-app visibility while respecting individual delivery preferences.

*Trace*: DR-14 (deduplicated alerting), DR-23 (Shared alert audience control)

*Milestone 3 source*: M3-QA-06.

*Verification*: Integration tests with repeated triggers, shared-household accounts, and different notification preferences.

---

=== MR-08: Barcode Scanning and Fallback

The mobile client shall use the device camera to scan barcodes for item identification. When barcode lookup fails, scanning is unavailable, or the product is unrecognized, the system shall immediately provide a manual-entry fallback requiring only item name and quantity/availability to save a minimum viable item record. Optional metadata shall remain editable later.

NOTE: Barcode uniqueness is a domain requirement, not a machine requirement. This MR addresses the interface/device aspect: using the camera to capture barcodes and falling back gracefully when scanning fails.

*Trace*: DR-1, DR-13, DR-21 (Minimal barcode fallback capture)

*Milestone 3 source*: M3-QA-04.

*Verification*: Manual tests with valid barcode, invalid barcode, unavailable scanner, and unrecognized local product scenarios.

---

=== Measurement Plan for Milestone 2 Completion

The following measurements will be collected during Milestone 2 validation activities to establish baseline for future milestones:

|===
|What to Measure |How |When

|Response times |Record during usability walkthroughs |Post-prototype
|Sync behavior |Verify real-time propagation via Supabase |During multi-user testing
|Error occurrences |Log all exceptions during testing |Throughout testing
|User perception |Participant feedback on speed/friction |During walkthroughs
|Barcode scanning success rate |Track match vs manual entry ratio |During walkthroughs
|=== 

These measurements will inform quantitative machine requirements for subsequent milestones.

---

=== Out of Scope for Milestone 2

Quantitative targets cannot be established without a prototype. These will be defined in Milestone 3 based on measurements collected during Milestone 2 validation.

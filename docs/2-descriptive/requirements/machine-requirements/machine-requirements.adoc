== Machine Requirements

Machine requirements define measurable quality properties for the implementation, together with the supported environment and deployment assumptions needed to run, test, and review the system. Milestone 2 established the initial quality properties (MR-01..MR-08) verified against the prototype, and Milestone 3 extends this section with the environment, platform, runtime, third-party service, and deployment requirements needed for end users, developers, and reviewers. Where specific thresholds are not yet known, the requirement defines what will be measured and how success will be evaluated.

Each requirement includes traceability to relevant Domain Requirements (DR) where applicable.

=== Supported Environment and Deployment Requirements

This section documents the minimum environment assumptions needed to run, test, and review the Digital Home Inventory system during Milestone 3. These requirements focus on the machine, platform, and external services needed for the application to operate consistently across development and review environments.

==== End-User Device Requirements

The system is intended to run as a mobile application. End users should have:

* A smartphone capable of running the project’s Flutter mobile client.
* A working device camera for barcode scanning features.
* Internet access for synchronization, authentication, shared household updates, alerts, and backend communication.
* Sufficient local storage for temporary cached data, pending offline changes, and local application state.
* A supported mobile operating system compatible with the Flutter version used by the project.

Barcode scanning and real-time synchronization may be limited or unavailable on devices without camera access, network connectivity, or required platform permissions.

==== Developer Machine Requirements

Developers and reviewers who need to run the project locally should have:

* Git installed for cloning and managing the repository.
* Flutter SDK installed and configured.
* Dart SDK available through the Flutter installation.
* A supported IDE or editor, such as Visual Studio Code or Android Studio.
* Android emulator, iOS simulator, or a physical mobile device for testing.
* Access to the project’s required backend configuration values when testing Supabase-connected features.
* Stable internet connection for dependency installation, Supabase access, and package downloads.

The main application source is located under the `src/` directory, which contains the Flutter project configuration.

==== Runtime and Framework Dependencies

The project depends on the Flutter/Dart mobile development stack. The current local development environment used for this documentation is:

* Flutter SDK: 3.35.5, stable channel
* Dart SDK: 3.9.2
* DevTools: 2.48.0

The Flutter project defines its Dart SDK constraint in `src/pubspec.yaml` as:

----
environment:
  sdk: ^3.8.1
----

Required package versions and Flutter dependencies are managed through `src/pubspec.yaml`, which is the authoritative dependency file for the mobile client.

At minimum, the local environment must support:

* Flutter project execution from the `src/` directory.
* Dart SDK compatibility with the `^3.8.1` SDK constraint.
* Dart package resolution using `flutter pub get`.
* Mobile build and run commands for the selected target platform.
* Device permissions required for camera-based scanning.
* Network access for backend communication.

Important project dependencies include:

* `supabase_flutter` for Supabase integration.
* `go_router` for navigation.
* `flutter_bloc`, `bloc_test`, and `mocktail` for state management and testing support.
* `get_it` for dependency injection.
* `intl` for formatting support.
* `pdf` and `printing` for report/export-related functionality.
* `fl_chart` for charting and visual summaries.

If dependency versions change, the `src/pubspec.yaml` file should be treated as the authoritative source for Flutter package requirements.

==== External Service Requirements

The system relies on external backend services for several core features. These services may include:

* Supabase database for persistent inventory, household, spending, and notification-related data.
* Supabase Auth for user identity and authenticated access.
* Supabase Realtime for shared household synchronization.
* Supabase Storage if item images, receipts, or related files are used.
* Supabase Edge Functions or scheduled backend logic for automation-related features, when enabled.
* Push notification services, such as FCM or APNs, if push delivery is enabled in later implementation stages.

Features that depend on these services may not work fully without valid project credentials, service configuration, and network connectivity.

==== Deployment Assumptions

For Milestone 3, deployment is treated as a review and testing environment rather than a production release. The project should be runnable locally by developers and reviewers who have access to the required dependencies and configuration.

Deployment-related assumptions include:

* The Flutter client can be run locally from the `src/` directory.
* Backend-connected features require valid Supabase project configuration.
* Environment-specific secrets or credentials should not be committed to the repository.
* Any deployment target used for review should be documented separately in the build and deployment procedure.
* If a production deployment target is not finalized, the documentation should clearly state that deployment remains a future or pending activity.

==== Environment Configuration

Environment-specific values, such as backend URLs, API keys, and service credentials, should be handled through local configuration files or secure project settings.

Configuration values may include:

* Supabase project URL.
* Supabase anonymous/public key.
* Platform-specific configuration for Android or iOS.
* Notification service configuration, if push notifications are enabled.
* Any feature flags or environment-specific settings required for testing.

Sensitive keys, private credentials, and service-role secrets must not be committed to version control.

=== MR-01: Availability

The system shall remain operational during supervised usability testing. No unplanned downtime requiring restart shall occur during testing sessions.

*Trace*: DR-12 (threshold-driven low-stock), DR-13 (expiration status)

*Verification*: Session logs; test coordinator observation.

---

=== MR-02: Response Time

The system shall respond quickly enough that household members can complete inventory tasks without waiting. During usability walkthroughs:
- No participant shall abandon a task due to perceived slowness
- All participants shall complete lookup tasks while simulating shopping (standing, holding phone)
- Item creation shall be completable in under 1 minute (consistent with US-INV-01)

Response time measurements will be collected during walkthroughs to establish baseline for future milestones.

*Trace*: DR-7 (short-window duplicates), DR-12 (threshold-driven low-stock)

*Verification*: Usability walkthrough observations; timing recorded for each task.

---

=== MR-03: Offline Operation

The mobile client shall support viewing inventory while offline and shall allow pending item creation and quantity updates while disconnected. Pending changes shall be clearly marked in the interface until synchronization succeeds. If the same tracked inventory record is changed remotely before synchronization, the system shall surface the conflict instead of silently overwriting data.

*Trace*: DR-9 (Auditability of corrections), DR-25 (Pending offline inventory changes)

*Milestone 3 source*: M3-QA-08.

*Verification*: Integration tests with connectivity simulation; manual tests creating and updating items offline, then reconnecting.

---

=== MR-04: Synchronization

When multiple household members use the same inventory, changes shall be propagated to all members in real time via Supabase Realtime. Changes shall appear without requiring manual refresh.

*Trace*: DR-4 (household administration), DR-5 (membership independence)

*Verification*: Automated tests verifying real-time delivery; manual tests with 2-3 concurrent users confirming changes appear instantly.

---

=== MR-05: Security and Privacy

Only authorized household members shall have access to household inventory data. Personal item records shall be hidden from non-owners by default unless explicitly shared. Household role permissions shall be enforced consistently across inventory, alerts, spending, and membership-management views.

*Trace*: DR-4 (household administration), DR-5 (membership independence), DR-6 (attribution retention), DR-18 (Personal item default privacy), DR-19 (Role-scoped household permissions)

*Milestone 3 source*: M3-QA-01, M3-QA-02.

*Verification*: Access control tests using owner/member/viewer/admin test accounts.

---

=== MR-06: Auditability

Quantity changes shall be logged with timestamp, user ID, item, and previous value.

*Trace*: DR-8 (ambiguity guard), DR-9 (auditability of corrections)

*Verification*: Schema validation; manual log inspection.

---

=== MR-07: Notification Quality

Repeated automated checks without meaningful state changes shall not generate duplicate alert records or repeated push notifications. Shared household alerts shall support household-level in-app visibility while respecting individual delivery preferences.

*Trace*: DR-14 (deduplicated alerting), DR-23 (Shared alert audience control)

*Milestone 3 source*: M3-QA-06.

*Verification*: Integration tests with repeated triggers, shared-household accounts, and different notification preferences.

---

=== MR-08: Barcode Scanning and Fallback

The mobile client shall use the device camera to scan barcodes for item identification. When barcode lookup fails, scanning is unavailable, or the product is unrecognized, the system shall immediately provide a manual-entry fallback requiring only item name and quantity/availability to save a minimum viable item record. Optional metadata shall remain editable later.

NOTE: Barcode uniqueness is a domain requirement, not a machine requirement. This MR addresses the interface/device aspect: using the camera to capture barcodes and falling back gracefully when scanning fails.

*Trace*: DR-1, DR-13, DR-21 (Minimal barcode fallback capture)

*Milestone 3 source*: M3-QA-04.

*Verification*: Manual tests with valid barcode, invalid barcode, unavailable scanner, and unrecognized local product scenarios.

=== Measurement Plan for Milestone 2 Completion

The following measurements will be collected during Milestone 2 validation activities to establish baseline for future milestones:

|===
|What to Measure |How |When

|Response times |Record during usability walkthroughs |Post-prototype
|Sync behavior |Verify real-time propagation via Supabase |During multi-user testing
|Error occurrences |Log all exceptions during testing |Throughout testing
|User perception |Participant feedback on speed/friction |During walkthroughs
|Barcode scanning success rate |Track match vs manual entry ratio |During walkthroughs
|=== 

These measurements will inform quantitative machine requirements for subsequent milestones.

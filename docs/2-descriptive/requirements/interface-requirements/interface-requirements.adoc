== Interface Requirements

Interface requirements describe how phenomena and concepts from the domain are represented inside the system-to-be, how they are initialized, and how they are kept in sync when the real world changes.

=== IR-01: Item Initialization
When a household member acquires a new item in the real world, they must have a way to create a corresponding representation inside the system.

*The system shall provide a means for household members to add new items to their tracked inventory, including name, quantity/availability, category, location, ownership status (personal/shared), and optional expiration date.*

**Questions this addresses:**
- How does an item first appear in the system?
- What information is needed to represent what was acquired?
- How does the system learn an item exists?

=== IR-02: Item Update on Consumption
When a household member consumes part of an item in the real world, the tracked representation must be updated to reflect new availability.

*The system shall provide a means for household members to record consumption by decreasing recorded quantity/availability or marking an item as finished/out-of-stock.*

**Questions this addresses:**
- How does the system learn an item was partially/fully consumed?
- How is tracked availability kept aligned with physical reality?
- What happens when an item is finished?

=== IR-03: Ownership Specification
In shared households, members need to know whether an item is personal or shared.

*The system shall provide a means to mark an item as personal or shared, and if personal, associate it with a household member.*

**Questions this addresses:**
- How does the system distinguish personal vs shared items?
- Who can see/act on which items?
- How are conflicts over personal items reduced?

=== IR-04: Low-Stock Threshold Definition
Households use different “low” definitions depending on the item.

*The system shall provide a means to set low-stock thresholds for individual items (with sensible defaults when no threshold is set).*

**Questions this addresses:**
- How does the system know what “low” means for different items?
- Can members customize thresholds to match their routines?
- What happens when no threshold is provided?

=== IR-05: Expiration Date Entry
Some items have expiration dates that must be captured to support expiration awareness.

*The system shall provide a means to enter and edit expiration dates for items that expire; items without expiration dates must be supported.*

**Questions this addresses:**
- How does the system learn when an item expires?
- Is expiration required or optional?
- How are items without expiration handled (no false confidence)?

=== IR-06: Purchase Logging
When a household member spends money on household items, that transaction must be representable in the system.

*The system shall provide a means to log purchases, including items bought (or categories), total cost, date of purchase, and whether the purchase was personal or shared.*

**Questions this addresses:**
- How does spending information enter the system?
- How can purchases be linked to items or categories?
- How is shared vs personal spending distinguished?

=== IR-07: Cost Splitting
For shared purchases, costs may need to be divided among household members.

*The system shall provide a means to split shared purchase costs among selected household members and track who owes what (according to household agreement).*

**Questions this addresses:**
- How are shared costs divided?
- How does the system know who participates in a split?
- How can disputes be reduced via clarity and traceability?

=== IR-08: Dashboard Metric Overview
Household members need a high-level overview without searching through details.

*The system shall provide a dashboard view summarizing key metrics such as total items, low-stock items, expiring/expired items, and current-period spending.*

**Questions this addresses:**
- How does a member quickly assess what needs attention?
- Which metrics matter most at a glance?
- How are different tracked phenomena summarized consistently?

=== Appendix A: Edge-Case Interface Requirements (Deferred)

The following edge-case interface requirements (IR-EC-A through IR-EC-I) are recognized but deferred for detailed specification in later milestones. They are captured in this appendix, separate from core IRs, to avoid losing scope awareness and to support future traceability.

==== IR-EC-A: Near-Duplicate Variant Disambiguation

When a user adds an item whose name matches an existing product but whose size, variant, or brand differs, the system must present the conflicting attribute (size, variant, flavor) highlighted and require the user to choose: "This is a different variant" or "Use existing product." Same-name/different-size entries shall be treated as separate product definitions by default and must not be auto-merged.

==== IR-EC-B: Stock Entry Separation by Expiration and Location

When a user attempts to merge two stock entries for the same product that have different expiration dates or different storage locations, the system must warn that merging will lose expiration/location tracking. The default action must be "Keep separate"; merging must require explicit confirmation.

==== IR-EC-C: Import Idempotency and Duplicate Spend Prevention

When a user uploads a CSV import file that was previously imported, the system must default to "Review before apply" and present a summary of potential duplicate transactions. Per-row, if a transaction fingerprint matches a previously imported record, the system must skip spend event creation by default and require an explicit "Import anyway" override.

==== IR-EC-D: Inventory Correction Without Budget Impact

When a user deletes or merges an inventory entry that has associated spend events, the system must present two independent controls: "Remove from inventory" (default on) and "Also mark purchase as mistake — exclude from budget" (default off). Inventory correction must never delete spend events automatically. Both actions must be logged separately in the audit trail.

==== IR-EC-E: Refund and Return Flow

When a user returns a purchased item, the system must provide a refund flow that: decrements inventory quantity, creates a negative spend entry linked to the original where possible, and requires amount refunded, date, and an optional link to the original receipt or spend event.

==== IR-EC-F: Waste and Shrinkage Adjustment

When a user removes expired, lost, or consumed items from inventory outside of a normal purchase/return flow, the system must: allow inventory adjustment without altering any spend event, require an adjustment reason, and offer an optional "Add to waste report" checkbox. Waste metrics must be reported separately from spending.

==== IR-EC-G: Shared Household Budget Attribution Prompt

When a user adds an item in a shared inventory, the system must prompt "Who bought this?" with the current user as default. If the user selects another household member, the system must require confirmation. Cross-user inventory merges are permitted for quantity counts, but each spend event must retain its original buyer attribution.

==== IR-EC-H: Undo for Auto-Merge Actions

Every auto-merge of inventory quantities (e.g., stock duplicate within 30-second window) must be reversible within 10 seconds via an undo control. Undo must revert quantity changes and record the undo in the audit log.

==== IR-EC-I: Audit Trail for Edge-Case Resolutions

Every edge-case resolution that modifies inventory state must generate a log entry containing at minimum: timestamp, user, household, source (manual / scan / import / adjustment / refund), action type, product reference, stock entry reference (if applicable), spend event reference (if applicable), reason (if user-selected), and a reference to the previous state. This log must be accessible to the household admin.

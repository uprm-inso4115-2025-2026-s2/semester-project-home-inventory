== Domain Requirements

Requirements expressed in domain language (capabilities/outcomes), independent of UI/technology.

=== Inventory Integrity
* DR-1: One barcode must map to exactly one Product definition. The system must not create a second Product definition for an already-known barcode.
* DR-2: A Stock entry must always reference a Product that existed at the time of the purchase.
* DR-3: Deleting or archiving an item must not erase its associated spending history — financial records are immutable.

=== Shared Household Rules
* DR-4: A Home must always have exactly one Owner. Ownership must be transferred before an Owner can leave.
* DR-5: A Stakeholder may belong to multiple Homes, each with an independent Role and Membership state.
* DR-6: Spending records must remain attributed to the Stakeholder who made the purchase, even when inventory quantities are merged.

=== Duplicate Handling
* DR-7: When a Stock Duplicate is detected within 30 seconds of the original entry, the system must aggregate quantity rather than create a new Stock row.
* DR-8: The system must not silently overwrite financial history when merging Stock entries — merge actions must be logged and reversible.

=== Budget & Spend Integrity
// NOTE (potential redundancy): DR-3 and DR-9 both state that financial records are immutable and must not be
// erased by inventory operations. DR-3 (Inventory Integrity) addresses deletion/archiving; DR-9 (Budget & Spend
// Integrity) addresses merge/split/adjustment. Consider merging them into one authoritative DR or clearly
// distinguishing the triggering conditions so readers understand why both exist.
* DR-9: Inventory state changes (merge, split, quantity adjustment) must never rewrite or delete Spend Events — financial records are immutable.
* DR-10: A Spend Event must remain attributed to the Stakeholder who initiated the purchase, even when shared inventory quantities are merged across household members.
* DR-11: Near-Duplicates must preserve user intent — the system must not auto-merge items whose product identity is ambiguous without explicit confirmation.
* DR-12: Every edge-case resolution that changes inventory state must be auditable and reversible; the system must log the action, the user, and the previous state.
* DR-13: Reports must distinguish between inventory state (counts, locations, expiry) and spend events (amounts, dates, attribution); these must not be conflated.

=== Prevention & Data Integrity
* DR-14: Duplicate prevention must occur before database insertion — the system must not create a new Product definition if an authoritative match (barcode) already exists.
* DR-15: Product definitions must be normalized before comparison; text-based matching must operate on canonical lowercase, trimmed, punctuation-free representations.
* DR-16: Spend Events must only be created after duplicate resolution is finalized — no spend record may reference an unresolved or provisional Product ID.
* DR-17: The system must guarantee that no two Products share an identical barcode; this constraint must be enforced at the database level.

=== Rationale / trace
* DR-1, DR-2, DR-3 support the need for accurate inventory and spending records (N-inventory-integrity)
* DR-4, DR-5, DR-6 support the need for clear household accountability in shared environments (N-sharing)
* DR-7, DR-8 support the need to prevent accidental duplicate entries while preserving auditability (N-data-quality)
* DR-9 through DR-13 support the need to protect budget accuracy and report integrity in shared household environments (N-budget-integrity)
* DR-14 through DR-17 support the need to prevent unintended duplicate records before they are persisted (N-data-quality)

== Domain Concept Analysis

This section derives domain concepts from the domain rough sketch (interviews, observations, edge situations). It documents how specific domain statements lead to abstractions (concepts), identifies potential term clashes, and defines relationships and invariants that hold in the domain independent of any system-to-be.

=== Starting point: raw statements held side-by-side

The following statements (from the rough sketch) are used as anchors for concept formation:

* Shared apartment: "If it is not on our fridge note, we assume we are out."
* Shared apartment: "Sometimes I check the pantry before leaving, but by the time I get to Walmart I forgot what we needed."
* Shared apartment: "We have rice in two places. One bag in the kitchen cabinet and another in a plastic bin in the hallway."
* Shared apartment: "Milk is usually bought twice in the same week at least once per month."
* Shared apartment: When asked how they know they are 'low': "when it feels light" or "when we open the backup." (and the backup opening is not recorded)
* Solo student: "If it takes more than a few seconds to log, I won't do it."
* Solo student: buys in bulk; yogurts sometimes expire; estimated 3–5 thrown away last semester.
* Household owner: child sometimes uses last item without saying; toilet paper runs out unexpectedly.
* Household owner: medicines kept separately; expiration dates important; thrown away expired medicine without realizing there was a second unopened box.
* Household owner: wants to remove access when someone moves out.
* Student athlete: consumption changes by context ("exam weeks"); buys based on memory of how fast it ran out last time.
* Brainstorming: people describe items as "full", "almost empty", "backup" (rarely with numbers).
* Edge situations: unreadable/missing expiration date; multiple expiration dates within a bulk package; backup stored in car trunk and forgotten.

These statements motivate the concepts below.

=== Concept formation: from statements to abstractions

We derive concepts by asking: (1) what phenomena exist in the domain, (2) what information households use to act, and (3) what changes over time.

Examples of abstraction:

* "If it is not on our fridge note, we assume we are out." ->
  *Tracked representation* (note/list) vs *reality* (actual item availability), and *assumption under uncertainty*.

* "Rice in two places" ->
  item storage *location*, and the possibility of multiple locations for the same item.

* "when it feels light" / "open the backup" ->
  *low-stock judgment* that is qualitative and strategy-based, and *backup* as a household practice.

* "Milk bought twice in the same week" ->
  *duplicate purchase* as a coordination failure, often arising from missing shared understanding.

* "3–5 yogurts thrown away last semester" / medicines expired with unopened box ->
  *expiration* and *batches* (same item type can exist in multiple packages with different expiration dates).

* "remove access when someone moves out" ->
  *membership* and *access to shared household information* (domain concept: who belongs to the household at a given time).

=== Core domain concepts (derived)

[cols="2,4",options="header"]
|===
|Concept |Definition (domain-level)

|Household
|A living unit (dorm, apartment, shared house) in which one or more people manage household essentials.

|Household member
|A person who currently belongs to a household and participates in consuming, replenishing, or coordinating around items.

|Household item
|A physical good kept for everyday use (e.g., milk, rice, toilet paper, soap, medicine).

|Item type
|The kind of item (e.g., "milk") independent of any specific package.

|Item instance / package
|A concrete package/unit of an item type (e.g., a specific milk carton, a specific box of medicine).

|Inventory
|The collection of household items that exist within a household at a given time.

|Tracked inventory record
|A representation of item availability used by household members (e.g., fridge note, mental model, list). This may diverge from the real inventory.

|Location (storage place)
|A physical place where items are stored (e.g., kitchen cabinet, hallway bin, bathroom closet, car trunk).

|Availability
|A household’s current “how much is left” understanding, often qualitative ("full", "almost empty", "backup").

|Quantity (optional formalization)
|A more precise measure of availability (counts/units). The domain can operate without exact numbers.

|Low-stock judgment
|A decision that an item is running low, derived from strategies like “feels light”, “opened backup”, or a numeric threshold.

|Backup supply
|A household practice of keeping a spare instance/package to reduce the risk of running out.

|Consumption
|Use of items over time (may vary by context such as exams/guests).

|Replenishment (restock)
|Acquiring more of an item after consumption reduces availability.

|Expiration date
|A date after which an item instance/package is considered expired or undesirable.

|Item batch (optional)
|A set of item instances with the same expiration date (useful when bulk packages contain multiple expirations).

|Coordination failure
|A situation where household members act on inconsistent information (e.g., duplicate purchase, unexpected shortage).
|===

=== Term clashes and clarifications

To prevent inconsistent usage across documentation, we clarify common ambiguous terms:

* *Inventory* (real-world items in the household) vs *Tracked inventory record* (note/list/memory). The rough sketch shows these diverge (e.g., “not on fridge note -> assume out”).
* *Item type* (“milk”) vs *Item instance/package* (a specific carton). This matters for expiration and “second unopened box”.
* *Low-stock* is not inherently numeric; in the domain it is often qualitative (“feels light”, “open backup”).
* *Backup* is a domain concept (household practice), not an application feature.
* *Household membership* changes over time (“remove access when someone moves out”): membership is a domain phenomenon; access control is a system concern derived from it.

=== Relationships (domain model)

[cols="2,2,4",options="header"]
|===
|Concept A |Concept B |Relationship

|Household |Household member
|A household has one or more members; membership may change over time.

|Household |Inventory
|A household has an inventory (items that exist in the household).

|Household item |Item type
|Each household item corresponds to an item type.

|Item type |Item instance/package
|An item type may have multiple item instances/packages in the household at once (bulk buying, backup supply).

|Item instance/package |Location
|Each item instance is stored in one location at a time; the same item type may be split across locations.

|Item instance/package |Expiration date
|Some item instances have an expiration date; others do not.

|Household member |Consumption / replenishment
|Members consume and replenish items; consumption rates vary by context (exam weeks, guests).

|Tracked inventory record |Inventory (reality)
|Tracked records approximate reality and may be incomplete, outdated, or wrong.

|Household |Coordination failure
|Coordination failures arise when members act on inconsistent or outdated tracked records.
|===

=== Events, actions, and behaviors (domain-level)

* Events (instantaneous state changes):
  - An item becomes *out* (unexpected shortage).
  - An item becomes *low* (as judged by a strategy such as “backup opened”).
  - An item expires (or is discovered expired).

* Actions (intentional acts):
  - Check availability (pantry check, fridge note check).
  - Consume an item (use the last roll, eat snacks faster during exams).
  - Replenish an item (buy milk, buy bulk yogurt).

* Behaviors (multi-step ongoing processes):
  - Household maintenance cycle: consume -> notice low/out -> decide purchase -> replenish -> store -> repeat.
  - Shared-household coordination: negotiate who buys what, or fail and duplicate-buy.

=== Invariants (domain rules)

These invariants describe stable properties of the domain:

* I1: A household has at least one member at any time it is considered active.
* I2: An item instance/package, if it exists, is stored in some location (even if the location is forgotten, e.g., car trunk).
* I3: Not all household judgments require numeric quantities; qualitative availability (“full”, “almost empty”, “backup”) is valid domain information.
* I4: If an item instance has an expiration date, it can become expired without being consumed, creating waste.
* I5: A household may possess multiple instances of the same item type simultaneously (bulk buying, backups).
* I6: Tracked records can diverge from reality; decisions may be made based on assumptions under uncertainty (“not on note -> assume out”).

=== Edge cases and risks (from rough sketch)

* E1: Same item type stored in multiple locations (rice in cabinet + hallway bin) -> risk of undercounting or duplicate purchase.
* E2: Backup supply opened but not recorded -> low-stock event is not propagated to shared understanding.
* E3: Consumption rate varies by context (exam weeks, guests) -> replenishment based on “last time” memory becomes unreliable.
* E4: Multiple expiration dates in a bulk package -> motivates batch-level tracking if expiration awareness is important.
* E5: Expiration date unreadable/missing -> household cannot reliably judge expiration status.
* E6: Membership change (move-out) -> household needs to update who participates and who is informed.

=== Ownership structures for shared inventory items

The rough sketch suggests that household items do not all behave the same way in shared environments. Some are treated as clearly shared essentials, while others are managed more personally even within the same household. For that reason, ownership must be analyzed as a domain concept rather than assumed to have only one structure.

Ownership structures are evaluated here based on their ability to:

- support collaborative interaction within a shared household context
- preserve clear responsibility attribution where needed
- align with domain invariants governing households, items, and membership
- support enforceable permission logic
- represent domain relationships coherently without unnecessary complexity

==== User-owned structure

In a user-owned structure, an item belongs primarily to an individual participant. Shared use is possible, but it must be added on top of individual ownership.

Relationship idea:

Item -> User +
Item associated with Household context

This structure provides strong individual accountability and clear ownership-based authority, but it does not fit shared-household semantics naturally. Collaboration becomes secondary, because the item is conceptually owned first by a person rather than by the shared household context.

This structure is most suitable when items are primarily personal contributions that only occasionally appear in a shared environment.

==== Group-owned structure

In a group-owned structure, an item belongs to the shared household context rather than to a specific person. Household members interact with the item through membership.

Relationship idea:

Household -> Item +
Household member -> Membership -> Household

This structure supports shared use naturally and aligns well with communal resources. However, it does not provide inherent responsibility attribution. If the domain needs to represent who is primarily responsible for a shared item, that information must be added separately.

This structure is most suitable when items are treated as collective household resources.

==== Hybrid structure

In a hybrid structure, an item exists within the shared household context but may also be associated with a responsible household member when needed.

Relationship idea:

Household -> Item +
Household member -> Membership -> Household +
Membership or responsibility relation -> Item

This structure preserves the shared household context while still allowing responsibility attribution. It supports both collaboration and accountability, though it is conceptually more complex than the other two structures.

This structure is most suitable when the domain must represent both shared items and items with a clearer responsible participant.

==== Recommendation

The hybrid structure provides the strongest overall conceptual fit for the current domain model.

It should be preferred because it:

- preserves the shared household context
- supports collaborative interaction naturally
- allows responsibility attribution when needed
- aligns with membership-based participation
- provides a flexible basis for later permission and ownership rules

This recommendation does not require every item to behave the same way. Instead, it provides a conceptual basis for representing both shared and personal responsibility inside one coherent household model.

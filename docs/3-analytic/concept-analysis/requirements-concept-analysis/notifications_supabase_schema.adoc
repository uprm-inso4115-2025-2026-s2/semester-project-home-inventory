= Design Supabase Database Schema for In-App Notifications & Alerts System
:toc:
:toclevels: 2

== Objective
Design the relational schema needed to support in-app alert / notification records for low-stock judgments, expiration monitoring, and shared household inventory events in the Digital Home Inventory application.

This document defines the database structures only. No triggers, scheduled jobs, or backend logic are implemented here.

== Proposed Solution
Define the core database structures:

* `notifications` table
* Enum types:
** `notification_type`
** `notification_status`
** `notification_priority`

The schema:

* Links alert / notification records to authenticated household member accounts (Supabase Auth)
* Optionally links alert / notification records to relevant item instances (packages) or tracked inventory records
* Supports alert / notification lifecycle state tracking
* Allows extensibility for additional alert / notification types

== Data Model

=== Table: `notifications`
[cols="1,1,3",options="header"]
|===
| Field | Type | Description
| id | uuid (PK) | Unique alert / notification record identifier
| recipient_user_id | uuid (FK → auth.users) | Household member account that receives the alert / notification
| inventory_item_id | uuid (nullable) | Optional reference to the relevant item instance (package) or tracked inventory record (FK to be added once the inventory schema is finalized)
| type | notification_type | Alert / notification type (LOW_STOCK, EXPIRING_SOON, etc.)
| title | text | Short alert / notification title
| message | text | Alert / notification message body
| priority_level | notification_priority | Alert / notification priority level
| status | notification_status | Alert / notification lifecycle state (UNREAD, READ, ARCHIVED, DISMISSED)
| created_at | timestamptz | Timestamp when the alert / notification record was created
| triggered_at | timestamptz | Timestamp when the underlying alert / notification condition was met
| expires_at | timestamptz (nullable) | Optional timestamp after which the alert / notification record is no longer relevant
| is_active | boolean | Indicates whether the alert / notification record remains active without deleting the row
|===

== Entity Relationships (Current State)

* `auth.users (1) -> (N) notifications`
** `notifications.recipient_user_id` references `auth.users.id`
** `ON DELETE CASCADE`

* Inventory relationship (Pending)
** `inventory_item_id` is nullable.
** A foreign key constraint will be added once the inventory table schema is defined.

== Supported Alert Scenarios (Documentation Only)

This section documents how the schema supports alert / notification creation (no trigger implementation required here).

* Low-stock alerts
** Trigger condition: `quantity <= threshold`
** Notification row:
*** `type = LOW_STOCK`
*** `inventory_item_id = <item_id>`
*** `status = UNREAD`

* Expiration alerts (expiring soon)
** Trigger condition: `expiration_date <= now() + X days`
** Notification row:
*** `type = EXPIRING_SOON`
*** `expires_at = <expiration_date>`
*** `status = UNREAD`

* Expiration alerts (expired)
** Trigger condition: `expiration_date <= now()`
** Notification row:
*** `type = EXPIRED`
*** `priority_level = HIGH` or `CRITICAL`
*** `status = UNREAD`

* Shared inventory alerts / notifications (roommates)
** Examples: shared item  instance updated/removed, duplicate detected
** Notification row:
*** `type = SHARED_INVENTORY_UPDATE` or `DUPLICATE_WARNING`
*** `inventory_item_id` optional depending on the event

== SQL Schema Draft (Supabase Compatible)

[source,sql]
----
-- =========================================
-- In-App Notifications & Alerts - Schema
-- =========================================

-- ENUM TYPES
create type notification_type as enum (
  'LOW_STOCK',
  'EXPIRING_SOON',
  'EXPIRED',
  'DUPLICATE_WARNING',
  'SHARED_INVENTORY_UPDATE'
);

create type notification_status as enum (
  'UNREAD',
  'READ',
  'ARCHIVED',
  'DISMISSED'
);

create type notification_priority as enum (
  'LOW',
  'MEDIUM',
  'HIGH',
  'CRITICAL'
);

-- TABLE
create table notifications (
  id uuid primary key default gen_random_uuid(),

  recipient_user_id uuid not null
    references auth.users(id)
    on delete cascade,

  inventory_item_id uuid null,

  type notification_type not null,
  title text not null,
  message text not null,

  priority_level notification_priority not null default 'MEDIUM',
  status notification_status not null default 'UNREAD',

  created_at timestamptz not null default now(),
  triggered_at timestamptz not null default now(),
  expires_at timestamptz null,

  is_active boolean not null default true
);

-- INDEXES
create index idx_notifications_recipient_created
  on notifications (recipient_user_id, created_at desc);

create index idx_notifications_recipient_status
  on notifications (recipient_user_id, status);

create index idx_notifications_type
  on notifications (type);

create index idx_notifications_inventory_item
  on notifications (inventory_item_id);
----

== Rationale

* Scalable: enum types enforce consistent alert categorization and allow adding new alert types without redesigning the table.
* Minimal redundancy: notifications store display-ready content and optionally link to inventory items.
* Lifecycle tracking: `status` supports transitions (UNREAD → READ → ARCHIVED/DISMISSED) without deleting data.
* Performance: indexes support common queries such as “recent notifications for a user” and filtering by status/type.

==== Concept Formation from Evidence and Artifacts

The 13 requirement-level concepts were formed by analyzing rough sketch statements, supporting research documents, and actor capabilities. The following table traces the originating evidence for each concept:

[cols="2,3,2",options="header"]
|===
| Requirement Concept | Evidence Source | Key Implication

| *Fast capture*
| "If it takes more than a few seconds to log, I won't do it"
| Interaction design must minimize burden (few fields, simple confirmation)

| *Tracked record vs reality*
| "I check the fridge note before buying"; "We have rice in two places." / household inventory changes continuously
| System represents availability at a point in time; divergence is inevitable and must be managed

| *Shared visibility*
| "Sometimes we buy the same item twice" / No formal responsibility / Reliance on fridge note + chat
| Multiple household members must converge on the same understanding without external coordination tools

| *Actionable states*
| Exam weeks/guest visits cause demand spikes; "toilet paper runs out unexpectedly"
| States must represent decision triggers, not just raw facts (NORMAL, LOW_STOCK, OUT_OF_STOCK, NEAR_EXPIRY, EXPIRED)

| *State transition*
| Repeated mentions of duplicates and alerts / Need to avoid notification fatigue
| Alert / notification events should trigger on *state transitions*, not on static conditions remaining unchanged

| *Low-stock strategy*
| Variable consumption / "runs out unexpectedly" / research in `low-stock-research-strategies.adoc`
| Fixed per item type low-stock threshold selected for balance between simplicity and flexibility (recommended in artifact)

| *Near-expiry window*
| "I don't notice when things expire" / research in `definition-expiration-alert-system.adoc`
| Definition of "expiring soon" (N days) required to alert before waste occurs; grace periods needed

| *Scheduled sweep (automation check)*
| Expiration and availability / low-stock concerns; need for monitoring beyond household member interactions / `automation-check-system.adoc`
| Daily background process evaluates low-stock and expiration conditions and generates alert / notification records for new state transitions

| *Notification record/event*
| Alerts required for expiration and low-stock / `evaluationOfNotification.adoc`
| Persisted representation of alerts (at least read/unread status in-app) needed for review and deduplication

| *Delivery approach*
| "I don't notice when things expire" (delayed awareness) / research in `evaluationOfNotification.adoc`
| In-app notification records selected for Milestone 1 (simpler, testable); push notifications remain optional

| *Deduplication / anti-spam*
| "Sometimes we buy the same item twice" (coordination failures) / need to avoid notification fatigue
| Rules required to prevent repeated alerts when state unchanged; repeated list additions when already recorded

| *Auto-add to restock list*
| Duplicate purchases + low-stock concerns; coordination through fridge note / `grocerylist_autoadd_decisionlogic.adoc`
| Rule-based triggers (OUT_OF_STOCK, EXPIRED, LOW_STOCK) convert actionable states into restock list entries

| *Transaction history*
| Tracking reliability / "I check the fridge note before buying" / `inventory-transaction-tracking.adoc`
| Append-only log of quantity changes with reason and timestamp enables traceability and audit; transaction history provides a traceable explanation for how the current quantity in the tracked inventory record was reached.
|===

==== Requirement Formation Decisions

The following requirement concepts were selected because they directly address domain problems identified in the rough sketch and supported by research artifacts:

- *Fast capture* — directly addresses "won't log if > few seconds" friction
- *Tracked record vs reality* — acknowledges that households are dynamic; system is a representation, not ground truth
- *Shared visibility* — necessary to reduce duplicate purchases and informal fridge-note coordination
- *Actionable states* — enables decision-making without requiring household members to interpret raw facts
- *State transition* — critical to alert / notification design; prevents notification fatigue
- *Low-stock strategy* — balances simplicity (fixed threshold) with flexibility (per-item type configuration)
- *Near-expiry window & scheduled sweep* — automate detection of waste risks
- *Notification record/event & delivery approach* — provide a way to notify without requiring push infrastructure
- *Deduplication / anti-spam* — protects against alert fatigue and repeated restock list additions
- *Auto-add to restock list* — extends alerts into actionable purchasing recommendations
- *Transaction history* — provides evidence trail for shared accountability and consumption analysis

==== Alignment with Domain Concepts

Requirement-level concepts align with domain concepts as follows:

- Fast capture — operations on tracked inventory records and item types / item instances
- Tracked record vs reality - gap between *Inventory (reality)* and the *tracked inventory record (system representation)*
- Shared visibility - *Household* (shared context) and interaction between *Household members*
- Actionable states - *Availability* (qualitative state: full, almost empty, backup) and derived judgment states
- State transition -triggering mechanism for *alert / notification* generation
- Low-stock strategy - implementation of *low-stock judgment* concept
- Near-expiry window - definition of *expiration-time concern*
- Scheduled sweep - systematic evaluation of *Item Instance expiration* and *Availability*
- Notification record/event - recording mechanism for system responses
- Delivery approach - infrastructure choice for household member communication
- Deduplication / anti-spam - rules governing *alert / notification* production
- Auto-add to restock list - connection between *Actionable states* and *purchasing intent*
- Transaction history - transaction history representing availability changes over time

This ensures consistency between domain analysis (Section 3.1) and requirements.

==== Scope Note

This evidence section traces all 13 requirement-level concepts to their originating rough sketch statements and supporting research artifacts. Concepts include both domain-grounded concerns (fast capture, shared visibility, actionable states) and system-design concerns (scheduled sweep, notification delivery, deduplication). While the latter may involve system behavior, they remain justified by domain drivers and supported by architectural research documented in the supporting artifacts referenced above.

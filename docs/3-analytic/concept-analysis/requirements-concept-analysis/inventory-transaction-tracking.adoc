= Inventory Transaction Tracking Model

== Key Information:

=== Issue ##148
=== Completed by: @Kemuel05

== Overview

This document defines how tracked inventory quantity changes should be recorded as transactions.
The goal is to keep a clear and consistent history of how and when item quantities change.

== What Is a Transaction (Quantity Change)

A transaction (quantity change) represents any action that changes the quantity of an item type (package) in the tracked inventory record.

Examples of transactions (quantity changes) include:

* Replenishment (restock) of an item type (quantity increases).
* Consumption of an item type (quantity decreases).
* Manual quantity adjustment of an item type.

If an action changes quantity, it should be considered a transaction.

== Transaction Data Table

This table describes the basic information each transaction should store.

[cols="1,1"]
|===
|Field |Description

|Item reference
|The item instance affected by the change.

|Quantity change
|The amount added or removed from the item’s quantity.

|Reason
|Why the change happened (restock, used, manual edit).

|Time of transaction 
|When the quantity change occurred.
|===

== Transaction Rules

* All quantity changes should be recorded as transactions.
* Transactions should never change quantity without a clear reason.
* Transaction history should be read-only after creation.

== Transaction Responsibility Boundary

Transactions (quantity changes) are responsible for recording quantity changes and updating tracked inventory record quantity.

Transactions may trigger immediate state evaluation after recording, but they do not directly own alert / notification creation or restock list / planner updates. Those actions occur only if the Automation System detects a state transition.

== Relationship to Inventory Quantity

* Tracked inventory records store the current quantity.
* Transaction (quantity change) records explain how that quantity was reached over time.
* Current quantity is updated when a transaction is recorded.
* Transactions are append-only historical records and are not the primary source used to compute current quantity during normal operation.

== Relationship Between Transactions and Low-Stock Judgment

After a transaction (quantity change) is recorded, the system may immediately evaluate low-stock judgment.
Low-stock judgment may also be evaluated during the scheduled sweep to ensure conditions are detected even when no recent transactions occurred.
If the transaction causes a state transition (e.g., NORMAL → LOW_STOCK or LOW_STOCK → OUT_OF_STOCK),
an alert / notification record may be generated and the item type may be added to the restock list / planner.


== Recommendation

Record all tracked inventory record quantity changes as transactions with a clear reason and timestamp.
This approach provides transparency and allows future features like history review and analytics.

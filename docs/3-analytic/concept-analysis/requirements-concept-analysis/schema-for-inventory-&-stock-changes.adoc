= Supabase Data Structure for Inventory and Stock Changes

== Key Information:

=== Issue #147
=== Completed by: @Kemuel05

== Overview

This section details a simple data structure for storing tracked inventory records and tracking quantity change transactions over time.
It focuses on the minimum data needed to support tracked inventory records and low-stock judgment logic

== Tracked Inventory Records Table

Represents the current state of each inventory item type.

[cols="1,1"]
|===
|Field |Description

|Item type name
|The name of the item type represented by this tracked inventory record.

|Category (optional)
|Optional grouping for the item type.

|Quantity
|The current quantity recorded in the tracked inventory record.

|Low-stock threshold
|The quantity at which the item type is considered low-stock.

|Created time (optional)
|When the tracked inventory record was created.
|===

== transaction Table

This table represents changes made to an item’s quantity over time.

[cols="1,1"]
|===
|Field |Description

|Tracked inventory record reference
|The tracked inventory record whose quantity was changed.

|Quantity change
|A positive or negative number representing the quantity change.

|Reason
|Why the quantity changed (replenishment, consumption, manual adjustment).

|Time of transaction
|When the quantity change occurred.
|===

== Relationship Between Tables

* One tracked inventory record can have multiple transaction records.
* transaction records explain how the current quantity of a tracked inventory record was reached.

== Quantity Ownership Rule

Tracked inventory records store the current quantity for each item type.
transaction records are append-only historical records that explain how the current quantity was reached over time.
When a transaction is recorded, the tracked inventory record quantity is updated accordingly.

== Simple Replenishment Flow

1. A household member replenishes an item type by adding quantity.
2. A transaction is recorded with a positive quantity value.
3. The tracked inventory record quantity is updated.
4. Low-stock judgment logic checks the updated quantity against the low-stock threshold.

== Schema Responsibility Boundary

This schema stores alert / notification records after a qualifying state transition has already been detected.

The schema itself does not evaluate low-stock, near-expiry, expired, or out-of-stock conditions. Those responsibilities belong to the Automation System.

== Recommendation

Using one table for tracked inventory records and one table for transaction records keeps the system simple and easy to extend.
This structure supports low-stock judgment while allowing future features like transaction history and availability tracking.

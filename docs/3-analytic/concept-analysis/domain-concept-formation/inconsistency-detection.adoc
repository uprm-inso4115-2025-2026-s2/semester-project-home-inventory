== Purpose

This document is the output of Stage 4 of the Domain Concept Formation Pipeline. It
catalogs all detected inconsistencies, conflicts, and incompleteness gaps found across
the raw domain description units (DDUs) and existing domain documents.

Each entry follows the pattern:

* *Type* — Direct P/not-P conflict, Scope conflict, Completeness gap, or Terminology drift.
* *Sources* — The DDU IDs or document sections where each side of the conflict appears.
* *Conflict statement* — The exact contradiction expressed as P and not-P.
* *Impact* — What downstream work is affected if left unresolved.
* *Resolution* — The recommended fix or the decision needed from the team.
* *Status* — Open / Resolved.

== Detected Inconsistencies

=== IC-01 — Spending Records: Immutable vs. Adjustable

[cols="1,3"]
|===
| Type | Direct P / not-P conflict
| Source P | `docs/4-specific-topics/domain-requirements-acquisition/` — Rule 5: "Spending events are immutable once recorded."
| Source not-P | `docs/2-descriptive/domain-description/functions/spending tracking logic/spending_tracking_logic.adoc` — Rule 7 / adjustSpending function: "Spending records can be corrected via an adjustment entry."
| P | A spending record cannot be changed after it is created.
| not-P | A spending record can be modified via an adjustment operation.
| Impact | Downstream budget calculations and audit trails depend on knowing whether historical records are authoritative or subject to revision.
| Resolution | Replace "immutable" with "append-only": original records are never deleted or edited in place; corrections are recorded as new *Expenditure* entries with a reference to the original. Both the original and the correction remain visible.
| Status | *Resolved* — adopt append-only model.
|===

=== IC-02 — Spending Persistence After Item Deletion

[cols="1,3"]
|===
| Type | Direct P / not-P conflict
| Source P | Spending/budget rules — "Deleting an item does not delete associated spending records." (spending history must persist for financial reporting)
| Source not-P | Data integrity rules — "No orphaned records are allowed; every spending entry must reference an existing item."
| P | Spending records survive item deletion.
| not-P | Spending records must always reference a live item record.
| Impact | If items are hard-deleted, spending records become orphans and violate referential integrity. If spending is deleted with the item, financial history is lost.
| Resolution | Use soft deletion: deleted items are marked `archived = true` but remain in the database. Spending records continue to reference the archived item row. Archived items are excluded from active inventory views but available to financial reports.
| Status | *Resolved* — adopt soft deletion.
|===

=== IC-03 — Household Ownership: Exactly One Owner vs. Multiple Owners

[cols="1,3"]
|===
| Type | Direct P / not-P conflict
| Source P | Domain requirement DR-4: "A Household has exactly one Owner at all times."
| Source not-P | Sharing rules document (`docs/descriptive/ownership models and rules/`): "The Owner role can be assigned to multiple roommates."
| P | A household must have exactly one Owner.
| not-P | The Owner role may be held by more than one member.
| Impact | Access control logic, permission checks, and the `household_members` schema depend on this cardinality. An unresolved conflict here will produce runtime permission errors.
| Resolution (Option A) | Rename the permission tier to "Admin" or "Co-manager", allow multiple admins, remove the "exactly one" constraint from DR-4.
| Resolution (Option B) | Keep a single designated Owner for legal/billing purposes; introduce a separate "Admin" role for operational access that multiple members can hold.
| Status | *Open* — decision required from team leads @FabianVelezOcasio and @andreasegarra.
|===

=== IC-04 — Low-Stock Judgment: Numeric Threshold vs. Qualitative Assessment

[cols="1,3"]
|===
| Type | Scope conflict
| Source P | Alert system requirements and Supabase schema — low-stock detection is defined as `quantity < threshold` (numeric).
| Source not-P | Domain narrative and terminology dictionary — "Low-stock is not inherently numeric; qualitative availability judgments are valid domain events."
| P | Low-stock is detected by comparing a numeric quantity against a stored threshold value.
| not-P | Low-stock can be judged qualitatively (e.g., "feels light", "only one left") without a numeric quantity.
| Impact | The system will miss valid low-stock events that users express qualitatively. Items without a recorded quantity will never trigger alerts even when genuinely low.
| Resolution | Support two detection modes: (a) automated numeric check when a quantity is recorded; (b) manual low-stock flag that a *Household Member* can set on any item regardless of recorded quantity. The *Availability State* concept must accommodate both.
| Status | *Open* — requires design decision on UI for manual flagging.
|===

=== IC-05 — Item Location: Always Known vs. Possibly Unknown

[cols="1,3"]
|===
| Type | Direct P / not-P conflict
| Source P | Entity model domain invariant I2: "An item instance, if it exists, is stored in some location (even if forgotten)."
| Source not-P | Domain narrative DDU-04: "Items may be forgotten in different locations" — implying the system may not have a recorded location.
| P | Every item has a location (physically true by definition).
| not-P | The system may not know an item's location (tracking gap).
| Impact | If the data model requires a non-null location reference, items discovered without a recorded location cannot be entered. This creates a data entry barrier and potential data loss.
| Resolution | Distinguish between the physical truth (an item always exists somewhere) and the tracked state (the system may record location as "Unknown"). The `location` field on an item instance should allow a sentinel value of `unknown` or `null` with a UI prompt to resolve it.
| Status | *Resolved* — allow null/unknown location in the schema with a prompt to update.
|===

=== IC-06 — Inventory Terminology: Reality vs. Record

[cols="1,3"]
|===
| Type | Terminology drift
| Source A | Entities document (Section 1, system-oriented): "Inventory — the collection of household items at a given time."
| Source B | Terminology dictionary: "Inventory (reality)" vs. "Tracked Inventory Record" — explicitly distinguished as two different concepts.
| Drift | The word "Inventory" is used in some documents to mean the real physical stock and in others to mean the system's representation of it. Downstream documents that inherit from Section 1 conflate the two.
| Impact | System designers building on the conflated definition will build a model where the tracked record is treated as ground truth, missing the key domain insight that it diverges from reality.
| Resolution | Enforce the two-name convention from the terminology dictionary across all documents: use *Inventory* only for the physical reality; use *Tracked Inventory Record* for what the system stores. Flag all occurrences of the word "inventory" that are not qualified.
| Status | *Open* — documentation team to audit and update affected sections.
|===

=== IC-07 — Alert Repetition: No-Spam Rule vs. Continuous State

[cols="1,3"]
|===
| Type | Scope conflict
| Source P | Terminology dictionary — "Alerts must not cause repeated notifications without a change in domain state."
| Source not-P | Alert system requirements — daily scheduled checks re-evaluate all items and emit alerts for any item still in a low/expired state, regardless of prior alerts.
| P | An alert fires only when state changes.
| not-P | The scheduled check fires alerts on every run for any item in an alert-worthy state.
| Impact | Users will receive repeated identical alerts for the same item day after day, causing notification fatigue and alert dismissal.
| Resolution | Add a "last alerted" timestamp to each alert record. The scheduled check only emits a new alert if (a) the state has changed since the last alert, or (b) a configurable re-notification interval (e.g., 7 days) has elapsed with no member action.
| Status | *Open* — requires implementation decision from Team 2 (@Solimar-Cruz).
|===

=== IC-08 — Budget Period End Date: Documented Requirement vs. Missing Domain Entity Field

[cols="1,3"]
|===
| Type | Direct P / not-P conflict
| Source P | `budget_management_logic.adoc` Section 3 — "Each budget period must store: Budget amount, Start date, *End date*, Remaining balance, Total spent, Status (Active, Completed, Exceeded)."
| Source not-P | `src/lib/features/sharing_budget/domain/entities/budget_entity.dart` — `BudgetEntity` fields are: `id`, `householdId`, `limitAmount`, `period` (weekly/monthly/yearly enum), `startDate`. No `endDate`, no `status`, no `remainingBalance`, no `totalSpent`.
| P | A budget period has a defined end date, a running total, and a status.
| not-P | The domain entity representing a budget period cannot compute or store its own end date, remaining balance, or status.
| Impact | Any feature that checks whether a spend event falls within a budget's active window (reporting, threshold alerts, remaining-balance display) must re-derive the end date from `period` + `startDate` ad hoc, with no shared rule. Teams will produce inconsistent results because the period boundary semantics are undefined — does `BudgetPeriod.monthly` starting 2026-01-15 end on 2026-02-14, 2026-02-15, or 2026-01-31?
| Resolution | The team must decide on one of two boundary interpretations and document it as a domain invariant: (A) *Rolling window* — end date is exactly N days after `startDate` (7 for weekly, 30 for monthly, 365 for yearly); (B) *Calendar alignment* — end date is the last day of the calendar week/month/year containing `startDate`. Once decided, `BudgetEntity` should expose a computed `endDate` and a `covers(DateTime date)` predicate derived from the agreed rule, so all features share a single source of truth.
| Status | *Open* — boundary interpretation decision required from @Gerardoruiz1.
|===

=== IC-09 — SpendEvent-to-Budget Linkage: Assumed vs. Undefined

[cols="1,3"]
|===
| Type | Completeness gap elevated to scope conflict
| Source P | `budget_management_logic.adoc` Section 5 — "Every spending record must be linked to an active budget period (if applicable). All relevant budgets must update in real time."
| Source not-P | `src/lib/features/sharing_budget/domain/entities/spend_event_entity.dart` — `SpendEventEntity` fields: `id`, `userId`, `householdId`, `productId`, `amount`, `date`, `source`, `receiptId`. No `budgetId`. `BudgetAllocationEntity` links a spend event to a user and split mode, but also has no `budgetId`.
| P | A spend event is linked to a budget period at the domain level.
| not-P | The domain entities provide no field or rule to establish this link. The only shared key between `SpendEventEntity` and `BudgetEntity` is `householdId`, which is insufficient when a household has multiple simultaneous budgets (e.g., household-level + category-level).
| Impact | Without an explicit linkage, the domain cannot answer "which budget(s) does this spend event affect?" consistently. Any budget-total calculation will either over-count (applying the same event to multiple budgets) or under-count (skipping events that should apply).
| Resolution | Define and document the domain rule: a `SpendEventEntity` contributes to a `BudgetEntity` if and only if `spendEvent.householdId == budget.householdId` AND `budget.covers(spendEvent.date)` (using the boundary rule resolved in IC-08). If category-level budgets are supported (per `budget_management_logic.adoc` Section 2.2), the rule must also match on spending category, which requires adding a `category` field to `SpendEventEntity`.
| Status | *Open* — depends on IC-08 resolution; category field decision required from @Gerardoruiz1.
|===

| `budget_management_logic.adoc`

| `SpendSource.refund` is defined as an enum value in the domain entities but the sign convention for refund amounts is never stated. It is unspecified whether a refund is recorded as a positive amount subtracted from total spend, or as a negative `amount` field on `SpendEventEntity`. Without this rule, refund handling will diverge across features (reports, budget totals, alerts).
| `src/lib/features/sharing_budget/domain/entities/enums.dart`, `src/lib/features/sharing_budget/domain/entities/spend_event_entity.dart`, `budget_management_logic.adoc`

== Domain Engineering

=== Domain Analysis and Concept Formation

The team applied domain engineering principles systematically by identifying and defining domain concepts directly from observations and interviews with real student users.

*Artifacts:*

* `docs/lecture-topic-tasks/domain_concept_formation.adoc` — Applies the concept formation method introduced in the Domain Engineering lectures. Raw domain description units (DDUs) were collected from students and then grouped and abstracted into three formal domain concepts: *Consumption Context*, *Distributed Stock*, and *Informal Household Role*. Each concept includes a formal definition and a domain narrative explaining how the concepts interact.
* `docs/lecture-topic-tasks/domainAttributesLTT.adoc` — Applies the Domain Attributes framework (Continuity, Discreteness, Hybridicity, Statics, Dynamics, Tangibility, and Dimensionality) to analyze the home inventory domain and inform system design decisions.
* `docs/lecture-topic-tasks/dimensionality_of_domain.adoc` — Identifies measurable dimensions for each domain phenomenon (e.g., item quantity, expiration time, location) to clarify the domain model.

=== Domain Assumptions

*Artifact:*

* `docs/lecture-topic-tasks/domain_assumptions.adoc` — Applies the domain assumptions framework from the lectures. The document explicitly identifies assumptions about user behavior (e.g., users will manually update quantities), data reliability, and system environment, and discusses their implications for system design. The document explains how unverified assumptions introduce risk and must be validated or revised.

=== Domain Inconsistency Detection

*Artifact:*

* `docs/lecture-topic-tasks/domain-inconsistency.adoc` — Applies inconsistency detection techniques. Two concrete inconsistencies were identified within the descriptive part: (1) the contradiction between spending records being labeled "immutable" and the presence of an `adjustSpending` function, and (2) the conflict between records persisting after item deletion and the requirement to avoid orphaned records. Each inconsistency includes a proposed resolution.
* `docs/4-specific-topics/inconsistency-management/` — Documents the broader strategy for managing inconsistencies across the project.

=== Static vs. Dynamic Classification

*Artifact:*

* `docs/lecture-topic-tasks/Static vs Dynamic Classification of Home Inventory Phenomena.adoc` — Classifies domain phenomena as static or dynamic (and dynamic subtypes: inert, reactive, programmable), directly applying the classification from the Domain Engineering module.

=== Human Behavior in the Domain

*Artifacts:*

* `docs/lecture-topic-tasks/Facets of Human Behavior in Home Inventory Management.adoc` — Applies the domain facets framework to analyze psychological and behavioral patterns (e.g., present bias, cognitive load, social loafing) relevant to student users and proposes design mitigations.
* `docs/lecture-topic-tasks/model_human_error_scenarios_ltt.adoc` — Applies the human error scenario modeling technique. Common error scenarios (duplicate entries, missed quantity updates, miscategorization, permission errors) are identified, their impact on the domain model is analyzed, and mitigation strategies are documented.

*Artifact:*

=== Scenarios and Walkthroughs

*Artifacts:*

* `docs/3-analytic/validation-verification/scenarios/` — Documents validation scenarios based on realistic student situations (e.g., a week in the life of a student managing a grocery budget). Scenarios are used to check whether planned features hold up against real student needs.

*Artifact:*

The testing strategy references specific user stories and requirements (e.g., US-INV-01, US-ALT-01) to link tests to requirements.

== Documentation Practices

=== Structured Documentation in AsciiDoc

The team applied the documentation practices emphasized in the course by writing all project documentation in *AsciiDoc* format, stored under `/docs` and organized into four parts:

* *Informative* (`docs/1-informative/`) — Current situation, needs, scope, team, and derived goals.
* *Descriptive* (`docs/2-descriptive/`) — Domain description, requirements, and implementation notes.
* *Analytic* (`docs/3-analytic/`) — Concept analysis, validation, and verification.
* *Specific Topics* (`docs/4-specific-topics/`) — Stakeholder identification, personas, traceability, domain requirements acquisition, and inconsistency management.

Documentation is versioned alongside source code in Git, making it traceable to specific milestones and issues.

=== Lecture Topic Tasks

The team produced dedicated *Lecture Topic Task* documents (`docs/lecture-topic-tasks/`) for each major course concept applied. These documents explicitly connect the lecture material to the project and are required per milestone, as documented in `CONTRIBUTING.md`:

> _"Minimum 1 lecture topic task per milestone."_

=== Style Guide and Contributing Guide

*Artifacts:*

* `STYLE_GUIDE.md` — Documents coding conventions and documentation standards, applying the knowledge management and documentation engineering practices covered in the course.
* `CONTRIBUTING.md` — Documents the full team workflow (issue creation → branch → PR → review → merge), applying Agile and collaborative software development practices discussed in class.

== Project Management and Tooling

=== Agile Workflow with GitHub

The team applied Agile development principles using GitHub as the project management platform:

* *GitHub Issues* — Used for task tracking, with structured templates aligned to course rubrics. Each issue includes a title, objective, proposed solution, success criteria, urgency rating, and difficulty rating.
* *GitHub Projects* — Used as a shared Kanban-style board to track work across feature teams.
* *GitHub Discussions* — Used for team communication and decision-making.
* *Pull Requests with mandatory review* — All changes to `main` require at least one reviewer who is not the author, implementing a peer review process consistent with quality assurance practices from the course.

=== Branch Strategy

The repository uses a structured branch strategy mapping to the team organization:

----
main (protected)
├── documentation/
├── design/
├── team-1-core-inventory/
├── team-2-alerts-restock/
├── team-3-sharing-budget/
└── team-4-reports-insights/
----

Each feature team works on their branch and creates PRs reviewed by team leads and managers before merging to `main`, applying separation of concerns and change management practices.

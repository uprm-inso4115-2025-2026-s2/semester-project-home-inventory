== Executive Summary

This document maps stakeholder groups to the key development phases of the Home Inventory Application. Stakeholder involvement changes throughout the software development lifecycl, since some stakeholders are critical early on for defining scope and requirements, while others become relevant only during implementation or testing. This analysis ensures that the right people are engaged at the right time, avoiding unnecessary meetings while ensuring critical input is captured when needed. The analysis covers three core development phases: Domain Engineering, Requirements Engineering, and Implementation, with justifications for stakeholder inclusion or exclusion at each stage.

== 1. Stakeholder Identification

Before mapping stakeholders to phases, we first identify all relevant stakeholder groups for the Home Inventory App:

[options="header"]
|===
| Stakeholder Group | Description | Key Interests
| End Users (Homeowners/Renters) | Primary users who will manage their personal belongings | Ease of use, quick item entry, reliable organization
| End Users (Roommates) | Users who need to track shared items and ownership | Item ownership tracking, shared categories
| End Users (Occasional Users) | Users who only access app for insurance claims or occasional updates | Simple interface, minimal learning curve
| Development Team | Developers implementing the application | Technical feasibility, clear requirements, maintainable code
| UX/UI Designers | Designers creating the user interface and experience | Usability, consistency, visual appeal
| Database Administrator | Manages data storage and structure | Data integrity, performance, scalability
| Security Specialists | Ensures application security | Data protection, authentication, authorization
| Third-Party API Providers | Provides barcode lookup, cloud storage, etc. | API limits, integration requirements
|===

== 2. Development Phases Overview

For this analysis, we focus on three core phases of the software development lifecycle:

[options="header"]
|===
| Phase | Description | Key Activities
| Domain Engineering | Understanding the problem domain, business context, and system scope | Market research, competitor analysis, feasibility studies, high-level vision
| Requirements Engineering | Eliciting, analyzing, specifying, and validating system requirements | User interviews, use case development, functional/non-functional requirements specification
| Implementation | Actual coding, development, and initial testing | Writing code, unit testing, integration, builds
|===

== 4. Phase 1: Domain Engineering

=== 4.1 Involved Stakeholders

[options="header"]
|===
| Stakeholder | Involvement Level | Justification
| End Users (All types) | ✅ Fully involved | Domain understanding requires direct input from users about their inventory management needs, pain points, and current workflows. Their context shapes the entire product vision.
| Development Team | ⚠️ Partially involved | Technical leads may be consulted for feasibility assessments, but the full team is not yet engaged.
| UX/UI Designers | ⚠️ Partially involved | May participate in early research to understand user workflows, but full design work begins later.
|===

=== 4.2 Excluded Stakeholders

[options="header"]
|===
| Stakeholder | Reason for Exclusion
| Database Administrator | Data schema cannot be designed without requirements.
| Security Specialists | Security requirements depend on feature scope not yet defined.
| Third-Party API Providers | API selection occurs after feature requirements are known.
|===

=== 4.3 Key Activities in Domain Engineering

- Conduct user interviews with homeowners, renters, and roommates

- Analyze competing inventory apps (sortly, home inventory, etc.)

- Define core value proposition: "Easy item tracking for insurance and organization"

- Identify initial feature set: barcode scanning, categories, photos, reports

- Assess technical feasibility of barcode scanning on mobile devices

- Define target platforms (iOS, Android, web)

== 5. Phase 2: Requirements Engineering

=== 5.1 Involved Stakeholders

[options="header"]
|===
| Stakeholder | Involvement Level | Justification
| End Users (All types) | ✅ Fully involved | Provide detailed input on features, workflows, and priorities through interviews, surveys, and prototype testing.
| Development Team | ✅ Fully involved | Provides technical input on feasibility, effort estimation, and implementation constraints.
| UX/UI Designers | ✅ Fully involved | Translate user needs into wireframes, prototypes, and interaction designs. Define user flows for adding items, browsing, and generating reports.
| Database Administrator | ⚠️ Partially involved | Consulted on data storage needs, relationships between items/categories/photos.
| Security Specialists | ⚠️ Partially involved | Review requirements for security implications (user accounts, data privacy, backup).
| Third-Party API Providers | ⚠️ Partially involved | Evaluate barcode lookup APIs, cloud storage options, and their integration requirements.
|===

=== 5.2 Excluded Stakeholders

[options="header"]
|===
| Stakeholder | Reason for Exclusion
| Implementation-Only Roles | Full development team not yet engaged in coding.
|===

=== 5.3 Key Activities in Requirements Engineering

- Develop user stories and use cases for core workflows:
** "As a homeowner, I want to scan item barcodes so I can quickly add them to my inventory"
** "As a roommate, I want to tag items with owner names so we can track shared purchases"

- Define functional requirements:
** Barcode scanning with manual fallback
** Category and room organization
** Photo capture and storage
** Search and filter capabilities

- Define non-functional requirements:
** Performance: Camera scans within 2 seconds
** Usability: Maximum 3 taps to add an item
** Security: User authentication, encrypted data storage
** Scalability: Support up to 10,000 items per user

- Create wireframes and prototypes for user validation

- Prioritize features for MVP vs. future releases

== 6. Phase 3: Implementation

=== 6.1 Involved Stakeholders

[options="header"]
|===
| Stakeholder | Involvement Level | Justification
| Development Team | ✅ Fully involved | Primary phase for developers to write code, implement features, and perform unit testing.
| Database Administrator | ✅ Fully involved | Implement data schema, optimize queries, manage data migration.
| Security Specialists | ✅ Fully involved | Conduct security reviews, penetration testing, and ensure secure coding practices.
| UX/UI Designers | ⚠️ Partially involved | Review implementation against designs, provide UI assets, and address visual inconsistencies.
| Third-Party API Providers | ✅ Fully involved | API integration requires direct work with external services (barcode databases, cloud storage).
|===

=== 6.2 Excluded Stakeholders

[options="header"]
|===
| Stakeholder | Reason for Exclusion
| End Users (All types) | End users are not involved in coding; they will be engaged later in user acceptance testing. Direct involvement during implementation would slow development.
|===

=== 6.3 Key Activities in Implementation

- Set up Flutter development environment

- Implement core features based on requirements:
** Barcode scanning screen with mobile_scanner
** Manual entry form with validation
** Database schema using SQLite or Firebase
** Category and room management
** Search and filter functionality
** Report generation

- Conduct weekly scrum meetings

- Document API integrations

- Create technical documentation

== 8. Key Takeaways

From this stakeholder mapping exercise, several important patterns emerge:

=== 8.1 Decreasing User Involvement
End users are heavily involved in domain engineering and requirements phases but are excluded from implementation. This is appropriate because:

- Users provide essential context about their needs and workflows

- Users validate that requirements match their expectations

- Users would slow down implementation with frequent change requests

- Users will re-engage during user acceptance testing (UAT) after implementation

=== 8.2 Increasing Technical Stakeholder Involvement
Technical roles (developers, database admin, security specialists) have increasing involvement as the project progresses:

- Minimal in domain engineering (feasibility only)

- Moderate in requirements (estimating, advising)

-mMaximum in implementation (execution)

=== 8.3 Third-Party Dependencies
API providers become relevant during requirements (selection/evaluation) and critical during implementation (integration).

== 9. Implications for Project Management

Understanding stakeholder involvement across phases enables:

. Clear Communication Plans: Know who needs updates at each stage
. Resource Planning: Allocate stakeholder time appropriately
. Risk Management: Identify missing stakeholder input early
. Expectation Setting: Prevent stakeholders from expecting involvement in inappropriate phases

== 10. Conclusion

This stakeholder mapping demonstrates that not all stakeholders participate in every development phase. End users are crucial for defining the problem and requirements but should not be involved during coding. Technical stakeholders become increasingly important as the project moves toward implementation.

By engaging the right stakeholders at the right time, the Home Inventory App project can:

- Capture accurate user needs

- Make efficient progress without unnecessary interruptions

- Ensure technical feasibility

- Deliver a product that meets both user expectations and established goals

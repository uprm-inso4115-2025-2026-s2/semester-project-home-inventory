== Summary

[horizontal]
Decision:: **fl_chart** recommended for all three chart types
Why:: Best balance of features, documentation, active maintenance, and MIT license
Migration Effort:: About 4-6 hours to replace all three chart implementations

This research evaluates different approaches for implementing dynamic bar, line, and pie charts in Flutter to replace the current static, hardcoded implementations in the Inventory Stock Summary, Item Usage Rates, and Expenditures report pages.

== Current State Analysis

The application currently uses three different custom implementations:

. *Expenditure Report Page* - Pie Chart
   - Implementation: `_PieChartPainter` (CustomPainter)
   - Data: Hardcoded `ExpenditureCategory` list
   - Features: Basic rendering, no tooltips or animations

. *Inventory Stock Summary Page* - Bar Chart
   - Implementation: Manual `Container`-based with dynamic heights
   - Data: Hardcoded `CategoryData` list
   - Features: Pagination support, basic bars, no tooltips

. *Item Usage Rates Page* - Line Chart
   - Implementation: `_LineChartPainter` (CustomPainter)
   - Data: Hardcoded `_kChartPoints` array
   - Features: Grid lines, data point dots, no touch interaction

=== Limitations Identified

- No dynamic data binding (static sample data only)
- Missing standard chart features (tooltips, legends, animations)
- Manual PDF capture quality depends on screen resolution
- No touch interaction (zoom, pan, data point selection)
- Three separate implementations to maintain

== Analysis of the Proposed Solutions

=== Option 1: Continue with CustomPainter

[cols="1,3", options="header"]
|===
| Type | Details
| Pros | 
Full control over rendering and behavior, No additional dependencies, Already works with current code, Smallest bundle size, Can optimize specifically for use case
| Cons |
Missing standard features (tooltips, animations, legends), More code to write and maintain, Manual PDF capture has quality limitations, No touch interaction support, Each chart type needs separate implementation
|===

[cols="1,3", options="header"]
|===
| Type | Details
| Pros |
Most popular Flutter charting library, Excellent documentation with many examples, Supports all chart types needed, Active development and community, MIT license (compatible with project), Built-in touch interactions and tooltips, Smooth animations out of the box

Adds ~500KB to bundle size, Some customization requires learning their API
|===

== Final Recommendation

After evaluating all options against the criteria, **fl_chart** is recommended for all three chart types.

=== Justification

[cols="1,4", options="header"]
|===
| Criteria | How fl_chart meets it
| Dynamic Data Binding | Excellent support with `setState` or stream updates
| Multi-Chart Support | Native support for bar, line, and pie charts
| PDF Export Quality | Can capture at any resolution using `RepaintBoundary`
| Active Maintenance | Recent commits (Dec 2024), responsive maintainer
| Learning Curve | Well-documented cookbook with copy-paste examples
| License | MIT - no restrictions for commercial use
| Performance | Handles 1000+ data points smoothly
|===

=== Why not continue with CustomPainter?

While custom implementation works for static data, adding dynamic data with features like tooltips, animations, and touch interactions would require significant development effort. fl_chart provides these features out of the box.

=== Why fl_chart over Syncfusion?

Syncfusion is feature-rich, but the commercial license and large bundle size make it less suitable. fl_chart provides all needed features with MIT license.

=== Chart-Specific Recommendation

|===
| Chart Type | Current Page | Recommended Library | Notes
| Bar Chart | Inventory Stock Summary | fl_chart | Replace manual Container bars
| Line Chart | Item Usage Rates | fl_chart | Replace CustomPainter implementation
| Pie Chart | Expenditure Report | fl_chart | Keep CustomPainter or migrate for consistency
|===

*Recommendation for Pie Chart*: Consider migrating to fl_chart as well for consistency across all chart implementations, though keeping the current CustomPainter is also acceptable given its simplicity.

=== Step 2: Bar Chart Implementation (Inventory Stock Summary)

Current implementation uses manual `Container` bars. Replace with:

[source,dart]
----
import 'package:fl_chart/fl_chart.dart';

class DynamicBarChart extends StatelessWidget {
  final List<CategoryData> data;
  
  const DynamicBarChart({required this.data});

  @override
  Widget build(BuildContext context) {
    return BarChart(
      BarChartData(
        barGroups: data.asMap().entries.map((entry) {
          return BarChartGroupData(
            x: entry.key,
            barRods: [
              BarChartRodData(
                toY: entry.value.quantity.toDouble(),
                color: AppTheme.primaryColor,
                width: 40,
                borderRadius: BorderRadius.circular(4),
              ),
            ],
          );
        }).toList(),
        titlesData: FlTitlesData(
          bottomTitles: AxisTitles(
            sideTitles: SideTitles(
              showTitles: true,
              getTitlesWidget: (value, meta) {
                return Text(data[value.toInt()].name);
              },
            ),
          ),
        ),
        gridData: FlGridData(show: true),
      ),
    );
  }
}
----

=== Step 3: Line Chart Implementation (Item Usage Rates)

Current implementation uses `_LineChartPainter`. Replace with:

[source,dart]
----
import 'package:fl_chart/fl_chart.dart';

class DynamicLineChart extends StatelessWidget {
  final List<double> points;
  final List<String> days;
  
  const DynamicLineChart({required this.points, required this.days});

  @override
  Widget build(BuildContext context) {
    return LineChart(
      LineChartData(
        lineBarsData: [
          LineChartBarData(
            spots: points.asMap().entries.map((entry) {
              return FlSpot(entry.key.toDouble(), entry.value);
            }).toList(),
            isCurved: false,
            color: Colors.black87,
            barWidth: 2,
            dotData: FlDotData(show: true),
          ),
        ],
        titlesData: FlTitlesData(
          bottomTitles: AxisTitles(
            sideTitles: SideTitles(
              showTitles: true,
              getTitlesWidget: (value, meta) {
                if (value.toInt() < days.length) {
                  return Text(days[value.toInt()]);
                }
                return const Text('');
              },
            ),
          ),
          leftTitles: AxisTitles(
            sideTitles: SideTitles(
              showTitles: true,
              interval: 10,
              reservedSize: 40,
            ),
          ),
        ),
        gridData: FlGridData(show: true),
        lineTouchData: LineTouchData(
          touchTooltipEnabled: true,
          touchCallback: (event, response) {
            // Handle tap for showing value
          },
        ),
      ),
    );
  }
}

=== Step 4: Pie Chart Implementation (Expenditure Report)

Current implementation uses `_PieChartPainter`. Replace with:

[source,dart]
----
import 'package:fl_chart/fl_chart.dart';

class DynamicPieChart extends StatelessWidget {
  final List<ExpenditureCategory> categories;
  
  const DynamicPieChart({required this.categories});

  @override
  Widget build(BuildContext context) {
    final total = categories.fold(0.0, (sum, c) => sum + c.amount);
    
    return PieChart(
      PieChartData(
        sections: categories.map((category) {
          final percentage = category.amount / total;
          return PieChartSectionData(
            value: percentage,
            title: '${(percentage * 100).round()}%',
            color: category.color,
            radius: 100,
            titleStyle: const TextStyle(
              fontSize: 12,
              fontWeight: FontWeight.bold,
              color: Colors.white,
            ),
          );
        }).toList(),
        sectionsSpace: 2,
        centerSpaceRadius: 40,
        startDegreeOffset: -90,
      ),
    );
  }
}
----

=== Step 6: PDF Export Enhancement

For higher quality PDF captures:

[source,dart]
----
Future<Uint8List?> _captureChart() async {
  final boundary = _chartKey.currentContext?.findRenderObject() as RenderRepaintBoundary?;
  if (boundary == null) return null;
  
  // Use higher pixel ratio for better PDF quality
  final image = await boundary.toImage(pixelRatio: 4.0);
  final byteData = await image.toByteData(format: ui.ImageByteFormat.png);
  return byteData?.buffer.asUint8List();
}
